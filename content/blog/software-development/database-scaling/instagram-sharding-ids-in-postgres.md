Most sharding war stories are really re-sharding war stories. A team picks a shard count, picks a hash, ships it, and then — eighteen months later when one box is on fire and another is idle — discovers that the only way to rebalance is to re-key half the dataset while the site is live. The pain was never the original sharding. The pain was that the sharding scheme couldn't *move*.

Instagram's early Postgres design is the canonical answer to that problem, and it is worth studying not because it is exotic but because it is almost boring once you see it. Two ideas do all the work. First: there are *thousands* of logical shards but only a handful of physical machines, so a shard is small enough to pick up and carry to a new box. Second: every primary key is a 64-bit integer that **encodes its own shard id in the middle 13 bits**, so given any row's ID you can compute which shard it lives on with a single bit-shift — no directory, no lookup table, no central service in the request path.

![The ID knows its own shard: decode by bit-shift, resolve one small map, query.](/imgs/blogs/instagram-sharding-ids-in-postgres-1.webp)

The diagram above is the mental model for the whole post. An ID arrives. You shift out 13 bits to recover its logical shard. You look up that logical shard in a tiny in-memory map to find the physical machine. You query. There is no per-row lookup table anywhere — the shard literally rode inside the integer. The rest of this article is a tour of how those two ideas fit together, how the IDs are generated by a stored procedure inside Postgres itself, why this beats both random UUIDs and a dedicated ID server, and where the scheme runs out of road.

## Why this design is different

The instinct, when you outgrow one database, is to "shard by user." That part Instagram did too. The non-obvious moves are what you do *around* the hash function. Here is the gap between the common mental model and what Instagram actually built.

| Question | Naive sharding | Instagram's Postgres sharding |
| --- | --- | --- |
| How many shards? | As many as you have machines (e.g. 8 DB boxes → 8 shards) | Thousands of *logical* shards (8192), on far fewer physical boxes |
| What is a shard, physically? | A whole database server | A Postgres **schema** inside a server; many schemas per box |
| How do you find a row's shard? | A lookup table keyed by user, or re-hash and hope | Bit-shift the row's own 64-bit ID; the shard is encoded in it |
| How do you generate IDs? | `SERIAL` per table, or a UUID, or a central ticket server | A PL/pgSQL function inside each schema that packs `time` + `shard` + `seq` |
| How do you rebalance? | Re-key data, change the hash, migrate rows | Copy a schema to a new box, update one line in the logical→physical map |
| What's the blast radius of "add a machine"? | Re-shard everything | Move some schemas; the sharding function never changes |

The pattern underneath every row of that table is the same: **separate the logical unit of partitioning from the physical unit of capacity, and make the routing key self-describing.** Once those two are separated, growth stops being a migration and becomes a `pg_dump | pg_restore` of a single schema plus a config change. If you have read [database partitioning and sharding](/blog/software-development/database/database-partitioning-and-sharding) and come away thinking "sharding is the part where everything gets hard," this is the design that makes the *operational* half stop being hard.

## 1. Logical shards versus physical machines

> Pick a logical shard count you will never outgrow, then forget it exists. Capacity is a property of machines; partitioning is a property of shards. Never tie the two together.

The single most important number in the whole design is the **logical shard count**, and the rule of thumb is: pick it once, pick it large, and pick a power of two. Instagram used **several thousand logical shards** — 13 bits of address space, so up to 8192 of them. Crucially, that number is frozen on day one and never changes again. It is *not* the number of database servers. On day one you might run those 8192 logical shards across three or four physical Postgres machines, with roughly 2700 schemas packed onto each box.

![A few physical machines, each holding thousands of logical schemas; a shard is a unit you can move.](/imgs/blogs/instagram-sharding-ids-in-postgres-3.webp)

Each logical shard is implemented as a Postgres **schema** — a named namespace inside a database that holds its own copy of the tables. Instagram named them `insta0`, `insta1`, …, so logical shard 1234 is literally the schema `insta1234`, and a query against it looks like `SELECT * FROM insta1234.photos WHERE ...`. Schemas are cheap: creating one is a catalog operation, scripting "create 8192 of them" is a loop, and `pg_dump --schema=insta1234` extracts exactly one shard's worth of data without touching its neighbors. That last property is the whole point. A schema is the granularity at which you can pick a shard up and carry it.

The mapping from logical shard to physical machine lives in application configuration — a small dictionary, a few kilobytes, easily cached in every app server's memory. It looks like this:

```python
# logical_shard_id -> physical machine. Thousands of shards, a handful of
# machines. This map is the ONLY thing that changes when you rebalance.
SHARD_TO_MACHINE = {
    # shards 0..2729 live on db01
    **{s: "db01" for s in range(0, 2730)},
    # shards 2730..5459 live on db02
    **{s: "db02" for s in range(2730, 5460)},
    # shards 5460..8191 live on db03
    **{s: "db03" for s in range(5460, 8192)},
}

CONNECTIONS = {
    "db01": "postgresql://app@db01.internal:5432/instagram",
    "db02": "postgresql://app@db02.internal:5432/instagram",
    "db03": "postgresql://app@db03.internal:5432/instagram",
}

def machine_for_shard(shard_id: int) -> str:
    return SHARD_TO_MACHINE[shard_id]
```

A useful way to feel the difference is to think about what an "add a machine" event touches. In a scheme where the shard *is* the machine, adding capacity means changing the hash function, which means every existing row now hashes to a different shard than the one it physically sits in — so you must physically move rows to match the new hash, and during the move the system is in a state where some rows are findable at the old location and some at the new one. That is the consistency minefield. In Instagram's scheme, the hash function `user_id → logical_shard` is *constant*; it produced shard 2732 last year and it produces shard 2732 today. Adding a machine never touches the hash; it only changes which machine *hosts* shard 2732. The set of rows in shard 2732 is identical before and after. Nothing is re-hashed, so nothing has to physically move except the one schema you chose to relocate.

The reason to make the logical count enormous up front is arithmetic. With 8192 shards you can grow from 1 machine to 8192 machines (one shard each, the theoretical ceiling) without ever changing the sharding function. In practice you will stop long before that, but the headroom means you never face the "we have 8 shards and 9 machines" problem, where the only fix is to re-shard. Choosing the shard key and shard count well is its own discipline; if you want the decision framework, see [choosing a shard key](/blog/software-development/database-scaling/choosing-a-shard-key). For this post, take the count as given: large, fixed, a power of two.

### Second-order: why a power of two

Thirteen bits of shard space is exactly 8192. The power-of-two choice is not aesthetic — it means extracting the shard from an ID is a mask (`& 0x1FFF`) rather than a modulo against an arbitrary number, and it means a `user_id → shard_id` mapping can be a cheap `user_id % 8192` (or a hash followed by a mask) that distributes evenly. The day you pick 8000 or 10000 instead of 8192, every shard extraction becomes a division and every off-by-one in your bit math becomes a silent data-routing bug.

## 2. The 64-bit ID: a number that carries its address

> A good distributed ID is three things at once: globally unique, roughly time-sortable, and self-routing. Instagram got all three out of 64 bits by spending each bit deliberately.

Here is the layout. A 64-bit signed bigint — the native, indexable, 8-byte integer type Postgres is happiest with — is divided into three fields:

- **41 bits of time**: milliseconds since a *custom epoch* (Instagram used `1314220021721`, representing the start of 2011). Forty-one bits of milliseconds is about 69 years of range, which from a 2011 epoch comfortably outlives the company's planning horizon. Using a custom epoch rather than the Unix epoch buys back the years 1970–2011 so the high bits aren't wasted.
- **13 bits of shard id**: the logical shard the row belongs to, 0 through 8191. This is the field that makes the ID self-routing.
- **10 bits of sequence**: a per-shard counter taken modulo 1024, so a single shard can mint up to 1024 distinct IDs within the same millisecond before it has to wait for the clock to tick.

Add them up: 41 + 13 + 10 = 64. Every bit is accounted for.

![The 64-bit ID splits into 41 bits of time, 13 bits of shard, and 10 bits of sequence.](/imgs/blogs/instagram-sharding-ids-in-postgres-2.webp)

The ordering of the fields is doing real work. Time goes in the **high** bits, which means when you sort IDs as integers you are sorting by creation time — `ORDER BY id` is `ORDER BY created_at` for free, and a `WHERE id > last_seen_id` becomes a time-based cursor with no separate timestamp column. The shard goes in the **middle**, recoverable with one shift and mask. The sequence goes in the **low** bits, where it tie-breaks IDs created in the same millisecond on the same shard without disturbing the global time ordering.

Here is the packing and unpacking in Python, which is the cleanest place to see the bit math before we move it into the database:

```python
EPOCH_MS = 1314220021721  # Instagram's custom epoch: ~2011-01-01, in ms

SHARD_BITS = 13
SEQ_BITS = 10
# field offsets, from the layout: [ time:41 | shard:13 | seq:10 ]
SHARD_SHIFT = SEQ_BITS                      # 10
TIME_SHIFT = SHARD_BITS + SEQ_BITS          # 23
SHARD_MASK = (1 << SHARD_BITS) - 1          # 0x1FFF, 13 ones
SEQ_MASK = (1 << SEQ_BITS) - 1              # 0x3FF,  10 ones

def make_id(now_ms: int, shard_id: int, seq: int) -> int:
    """Pack a 64-bit, time-sortable, shard-encoding id."""
    assert 0 <= shard_id < (1 << SHARD_BITS)
    ms = now_ms - EPOCH_MS
    return (ms << TIME_SHIFT) | (shard_id << SHARD_SHIFT) | (seq & SEQ_MASK)

def shard_of(id_: int) -> int:
    """Recover the logical shard from any id — the routing primitive."""
    return (id_ >> SHARD_SHIFT) & SHARD_MASK

def timestamp_ms_of(id_: int) -> int:
    """Recover the creation time in ms — useful for range scans and TTLs."""
    return (id_ >> TIME_SHIFT) + EPOCH_MS

def seq_of(id_: int) -> int:
    return id_ & SEQ_MASK

# A concrete row: created in shard 1234, first id of its millisecond.
photo_id = make_id(now_ms=1459220021721, shard_id=1234, seq=0)
print(photo_id)               # 1216348160001263616
print(shard_of(photo_id))     # 1234   <- no lookup table touched
print(seq_of(photo_id))       # 0
print(timestamp_ms_of(photo_id))  # 1459220021721, the original ms
```

Run it and `shard_of(1216348160001263616)` returns `1234`. That call is the entire routing layer. There is no network round-trip, no table, no cache to warm — just a shift and an AND that any language can do in nanoseconds. The shard the row lives on has been baked into its identity since the moment it was created.

### Second-order: the sequence is per shard, not global

A subtle point that trips people up: the 10-bit sequence is *not* a global counter. It is per-shard, taken modulo 1024. Two different shards can both mint an ID with `seq = 7` in the same millisecond and there is no collision, because their shard fields differ. The uniqueness guarantee is the *combination* of (time, shard, seq): within one shard, time + seq is unique up to 1024 IDs/ms; across shards, the shard field separates everything. This is why the sequence only needs 10 bits — it never has to be globally unique, only unique within (shard, millisecond). If a single shard ever needs more than 1024 IDs in one millisecond, the generator blocks until the clock advances, which at Instagram's per-shard write rates was never close to a real constraint.

## 3. Generating the ID inside Postgres, per schema

> The best place to mint an ID is right next to the data it identifies. Instagram put the generator inside each shard's schema as a PL/pgSQL function, so the database hands you a correct ID on insert and there is nothing else to call.

The Python above is for understanding. In production the ID is generated by a stored function written in PL/pgSQL — Postgres's built-in procedural language — and there is **one copy of that function inside every schema**, with that schema's shard id baked in as a constant. This is the actual function Instagram published, lightly annotated:

```sql
CREATE OR REPLACE FUNCTION insta5.next_id(OUT result bigint) AS $$
DECLARE
    our_epoch  bigint := 1314220021721;  -- custom epoch, ms (~2011-01-01)
    seq_id     bigint;
    now_millis bigint;
    shard_id   int := 5;                  -- THIS schema's shard id, a constant
BEGIN
    -- 10-bit per-shard sequence: a real Postgres sequence, taken mod 1024.
    SELECT nextval('insta5.table_id_seq') % 1024 INTO seq_id;

    -- wall clock in milliseconds (clock_timestamp = real time, not tx start)
    SELECT FLOOR(EXTRACT(EPOCH FROM clock_timestamp()) * 1000) INTO now_millis;

    -- pack: [ (ms - epoch) << 23 | shard_id << 10 | seq_id ]
    result := (now_millis - our_epoch) << 23;
    result := result | (shard_id << 10);
    result := result | (seq_id);
END;
$$ LANGUAGE PLPGSQL;
```

![next_id() reads the sequence, the clock, and the constant shard, then ORs three shifted fields into one bigint.](/imgs/blogs/instagram-sharding-ids-in-postgres-7.webp)

Three things make this elegant. First, `shard_id` is a literal `5` because this function lives in the `insta5` schema — every schema gets its own copy with its own constant, so the shard is determined by *which schema you insert into*, not by any parameter the application has to remember to pass. Second, the per-shard counter reuses Postgres's existing `nextval()` machinery — a real sequence object, `insta5.table_id_seq` — which is already concurrency-safe and crash-safe, so Instagram wrote zero new locking code. Third, `clock_timestamp()` (not `now()` or `CURRENT_TIMESTAMP`, which return the transaction start time) is used so two rows in the same transaction still get monotonically increasing time bits.

You wire it into a table by making it the column default:

```sql
-- inside schema insta5
CREATE SEQUENCE insta5.table_id_seq;

CREATE TABLE insta5.photos (
    id     bigint     NOT NULL DEFAULT insta5.next_id(),
    PRIMARY KEY (id),
    user_id    bigint NOT NULL,
    created_at timestamptz NOT NULL DEFAULT now(),
    caption    text
);
```

Now an `INSERT INTO insta5.photos (user_id, caption) VALUES (...)` returns a freshly packed, shard-stamped, time-sortable `id` with no application involvement at all. The database is the source of truth for IDs, the same way it is the source of truth for the rows.

### Second-order: why generation at the data, not at the edge

It would be tempting to generate the ID in the application and pass it in. Instagram deliberately did not, and the reason is correctness under concurrency. The 10-bit sequence has to be atomic per shard — if two app servers both decided "this is seq 7 in shard 5 at millisecond M," you'd get a duplicate key. Postgres's `nextval()` already guarantees atomic, gap-tolerant increments across all concurrent transactions; reusing it means the uniqueness guarantee is enforced by the database that also enforces the primary key, not by hopeful coordination between stateless app servers. Generate the ID where the constraint lives.

## 4. The read path: routing with a bit-shift

> Routing is not a lookup, it is a computation. Given an ID, the shard is `(id >> 10) & 0x1FFF`. Everything after that is resolving one small, cacheable map.

When a client comes back holding a photo ID and asks "give me this photo," the application does not consult any per-row directory. It does this:

```python
def route(id_: int) -> tuple[str, str]:
    """Map any row id to (physical machine, schema) with no I/O."""
    shard_id = (id_ >> 10) & 0x1FFF          # extract 13 shard bits
    machine = machine_for_shard(shard_id)    # tiny in-memory dict
    schema = f"insta{shard_id}"
    return machine, schema

def fetch_photo(id_: int):
    machine, schema = route(id_)
    conn = get_connection(CONNECTIONS[machine])
    return conn.execute(
        f"SELECT * FROM {schema}.photos WHERE id = %s", (id_,)
    ).fetchone()

# fetch_photo(1216348160001263616) -> shard 1234 -> db07 -> insta1234.photos
```

![The write path packs the shard into the id; the read path shifts it back out to find the row.](/imgs/blogs/instagram-sharding-ids-in-postgres-4.webp)

Trace the figure end to end. On the **write path**, the application hashes the `user_id` to a shard, inserts into that shard's schema, and `next_id()` stamps the shard into the returned ID. On the **read path**, the shard is shifted back out of that same ID, resolved through the in-memory map to a machine, and queried directly. The shard makes a complete round trip *inside the integer itself* — packed in on write, shifted out on read. The only piece of state the read path needs is the logical→physical map, which is a few kilobytes and changes only when you rebalance.

Compare this to the alternative most teams reach for first: a `lookups` table that maps `photo_id → shard_id`, or worse `photo_id → machine`. That table has to be consulted on every read, has to be sharded itself (it grows with your data), has to be kept consistent, and becomes a hot path that needs its own caching tier. Instagram has none of that. The "lookup table" is replaced by a shift instruction.

### Second-order: routing the write when you only have a user

There is an asymmetry worth naming. On reads you have the row's ID, so the shard falls out by bit-shift. On writes — say, "create a photo for this user" — you don't have a row ID yet, so you compute the shard from the **shard key**, which for Instagram is the `user_id`: `shard_id = user_id % 8192` (or a hash of it, masked to 13 bits). You then insert into that shard's schema, and `next_id()` records the shard you chose into the new ID. From then on, every read of that row routes by bit-shift. The shard key picks the shard once, at creation; the ID remembers it forever.

## 5. Growing without re-keying: move a schema

> The whole design exists to make one operation cheap: relieving a hot machine. You do it by moving schemas, not rows, and the IDs never change.

Eventually a physical machine runs hot — too many writes, too much disk, a noisy shard. In a naive scheme this is the catastrophe: you change the hash, and now existing rows hash to different shards than where they physically live, so you must migrate data *and* the migration window is a consistency minefield. In Instagram's scheme it is a maintenance task.

![Relieving a hot machine moves one schema to a new box and rewrites one line of the map; ids are unchanged.](/imgs/blogs/instagram-sharding-ids-in-postgres-5.webp)

The procedure: stand up a new physical machine, pick some logical schemas off the hot box (ideally a hot one), copy each schema to the new box, then flip those schemas' entries in the logical→physical map to point at the new machine. Concretely:

```bash
# 1. Move logical shard 2732 (schema insta2732) from the hot db02 to a new db04.
pg_dump --schema=insta2732 \
        --host=db02.internal --dbname=instagram \
  | psql --host=db04.internal --dbname=instagram

# 2. (Replication/CDC keeps the new copy current during the dump; then a brief
#    cutover freezes writes to insta2732, applies the tail, and switches over.)
```

```python
# 3. The only application change: one line in the map.
SHARD_TO_MACHINE[2732] = "db04"   # was "db02"
# Every id that encodes shard 2732 is UNCHANGED. No re-keying. No re-hashing.
```

The reason the IDs never change is exactly because the shard id was frozen into them at creation. Shard 2732 is still shard 2732; only its *physical address* moved. A photo created two years ago with shard 2732 in its ID still routes correctly the instant the map flips — the bit-shift still yields 2732, and 2732 now resolves to `db04`. There is no window where old IDs point at the wrong place, because old IDs don't point at a *place* at all; they point at a *logical shard*, and the logical shard's address is the only thing that moved.

This is the same family of technique as Vitess's `MoveTables` and modern resharding tooling; if you want the live, zero-downtime mechanics of streaming a shard to a new home, see [live resharding and rebalancing without downtime](/blog/software-development/database/live-resharding-and-rebalancing-without-downtime). The Instagram version is simpler precisely because the granularity is a whole schema and the routing key never has to be rewritten.

### Second-order: the granularity tradeoff

Moving happens at schema granularity, and that is both the strength and the limit. Strength: a schema is a clean, self-contained unit — `pg_dump --schema` gives you exactly its rows and nothing else, and the cutover affects only the users who hash to that one shard. Limit: a *single* schema is the smallest thing you can move. If one schema is itself too hot — one shard with a celebrity user generating a huge fraction of writes — you cannot split it further without re-keying, because everything in `insta2732` shares the same 13-bit shard field. The fix is to choose a shard key with good distribution up front (so no single shard concentrates load) and to pick a logical shard count large enough that each shard stays small. Granularity is decided at design time, in the shard count.

## 6. Why not random UUIDs

> A v4 UUID is the path of least resistance for a unique key and the path of maximum resistance for the database underneath it. Random keys are an index's worst case.

The reflexive alternative to a custom ID scheme is "just use a UUID." It is globally unique, generatable anywhere, no coordination. So why didn't Instagram? Two concrete reasons, and the second is the one that matters most at scale.

The first is **storage**. A random UUID is 128 bits. Instagram's ID is 64. Double the width on the primary key, doubled again on every foreign key and every secondary index that references it, across billions of rows, is a large and permanent tax on disk, memory, and cache.

The second, and the real killer, is **index locality**. A v4 UUID is random, so successive inserts land at random positions in the primary-key B-tree. A time-sorted ID is monotonic, so successive inserts land at the *right edge* of the B-tree, in the same leaf page, over and over.

![Random keys dirty leaf pages all over the B-tree; time-sorted keys keep writes appending to one hot right-edge page.](/imgs/blogs/instagram-sharding-ids-in-postgres-6.webp)

The difference is brutal under write load. With random UUIDs, each insert touches a different, cold leaf page: it must be read from disk (cache miss), modified, and written back, and as pages fill, random inserts trigger expensive page splits all over the tree. Your working set for writes is effectively the *entire* index, which does not fit in RAM. With time-sorted IDs, every insert appends to the rightmost leaf — one hot page that stays pinned in the buffer cache, almost never splits, and writes back sequentially. The write working set is a single page. This is the same B-tree fragmentation story told in detail in [random UUIDs are killing your database performance](/blog/software-development/database/random-uuids-are-killing-your-database-performance) — Instagram's design is, in effect, the cure for that disease applied before the disease could take hold.

| Property | Random UUID v4 | Instagram 64-bit id |
| --- | --- | --- |
| Width | 128 bits | 64 bits |
| Sortable by creation time | No (random) | Yes (time in high bits) |
| Insert locality in B-tree | Scatters across all leaves | Appends to one hot leaf |
| Page splits / write amplification | High | Minimal |
| Encodes its own shard | No (needs a separate map) | Yes (13 middle bits) |
| Index working set under writes | Whole index | One page |

There is a category of "sortable UUID" (ULID, UUIDv7) that fixes the locality problem by putting a timestamp in the high bits — and those are excellent modern choices. But they are still 128 bits, and they still don't encode a shard. Instagram needed both the locality *and* the self-routing in a native 64-bit integer, and the only way to get all three was to design the field layout themselves.

## 7. Why not a central ID server

> A dedicated ID service removes coordination from the database and adds it to the network. You trade a clean bit-shift for an RPC and a new thing that can be down.

The other well-known approach is a dedicated ID-generation service — Flickr's "ticket server," Twitter's Snowflake. A central service (or a small cluster) hands out unique IDs on request. Snowflake in particular uses a very similar bit layout (timestamp + machine + sequence), so the *ID format* idea is shared. The difference is *where the generation happens*.

Instagram considered this and rejected it on operational grounds: **the additional complexity of running an ID service was a point against it.** Unpack that and there are three concrete costs:

1. **A network hop on the write path.** Every insert now needs a round-trip to the ID service before it can write. That is latency on the hot path and a dependency on a remote system being reachable.
2. **A new single point of failure.** If the ID service (or its cluster) is unavailable, you cannot create anything, anywhere. You have coupled the availability of every write in the system to the availability of one more component. Generating IDs inside each shard's own Postgres means an ID can be minted exactly when and where the row is written — if that shard's database is up, IDs work; if it's down, you couldn't write to it anyway.
3. **More to operate.** A separate service is another deployment, another on-call surface, another thing to capacity-plan, monitor, and patch. Instagram's generator is a stored function — it ships with the schema, scales with the database, and has no independent failure mode.

![Three ways to mint a distributed id, scored on storage, sort, hops, single points of failure, and routing.](/imgs/blogs/instagram-sharding-ids-in-postgres-8.webp)

The matrix lays out the three-way comparison. Read down the rightmost column: the embedded-shard ID is the only approach that is simultaneously narrow (64 bits), naturally time-sorted, free of an extra write hop, free of a new single point of failure, *and* self-routing with no lookup. Snowflake-style central servers match it on width and sort but lose on the hop and the SPOF; UUIDs lose on width, sort, and routing. Instagram's insight was that they didn't have to choose between Snowflake's nice ID format and Postgres's local generation — they could have the format *and* keep generation inside each database, by writing the same bit-packing logic as a PL/pgSQL function.

### Second-order: clock skew is the price you pay

The honest tradeoff of generating time-based IDs in many independent places is that each place trusts its own clock. If a machine's clock jumps backward (NTP correction, VM migration), `next_id()` could in principle produce an ID with a *smaller* time prefix than one it just generated, breaking strict monotonicity for that shard. In practice the 10-bit sequence absorbs small skew within a millisecond, IDs only need to be roughly time-sortable (not strictly), and uniqueness is still guaranteed by the (shard, seq) combination even if the time bits wobble. A central server sidesteps clock skew — at the cost of the hop and the SPOF above. Instagram judged "roughly sortable, clock-trusting, no SPOF" the better deal, and for a photo feed it clearly is.

## Case studies from production

These are illustrative scenarios — the kinds of incidents this design shapes — drawn from how the scheme behaves under real operational pressure. Names are generic; the mechanisms are exact.

### 1. The hot machine that wasn't a hot shard

A physical box, `db02`, starts showing write latency and disk pressure. The first hypothesis is always "a hot shard" — one schema taking disproportionate traffic. But the metrics show writes spread evenly across all ~2700 schemas on `db02`; the box is simply at capacity for the *aggregate* of its shards. The fix is not to split any shard but to **redistribute**: stand up `db04`, move 1300 of `db02`'s schemas to it with `pg_dump --schema`, and flip those 1300 map entries. Aggregate write load on `db02` halves. No ID changed, no application code beyond the map. The lesson: separating logical shards from physical machines means "machine is full" and "shard is hot" are *different* problems with different fixes, and the cheap one (redistribute schemas) handles the common case.

### 2. The celebrity shard

Years in, one shard — `insta2732`, holding a handful of accounts that went viral — generates writes far out of proportion to its neighbors. Now moving the schema to its own dedicated box helps with the *machine* being shared, but the shard itself is the bottleneck and **cannot be split**, because every row in it carries `2732` in its 13 shard bits and re-keying would change every ID. The team's only levers are application-side: cache that shard's reads aggressively, move some of its write-heavy derived data (counters, feeds) into a different store, and resolve never again to let a shard key concentrate this badly. The lesson is the granularity limit from §5, met in the wild: the smallest unit you can move is a whole schema, so the shard key and shard count are load-bearing decisions you make on day one and live with forever.

### 3. The lookup table that never had to exist

A new engineer, building a feature that needs to find every comment on a photo, instinctively reaches for a `comment_id → shard` index table — the pattern from their last job. Code review catches it: the comment's shard is already in its ID. `(comment_id >> 10) & 0x1FFF` is the entire "index." The proposed table would have been a write on every comment, a read on every lookup, sharded itself, and a cache tier — all to store information that was already encoded in the primary key. The lesson: when your IDs are self-routing, the reflex to build a routing directory is a smell. The map you need is logical→physical (kilobytes), not row→shard (terabytes).

### 4. The clock that ran backward

A Postgres host is live-migrated between hypervisors and its clock briefly jumps backward by 40 milliseconds. For those 40 ms, `next_id()` on that shard produces IDs whose time prefix is *lower* than IDs minted just before the jump. Nothing breaks: the IDs are still unique (the sequence and shard fields guarantee it), and the only visible effect is that a `ORDER BY id` over that narrow window is very slightly out of true creation order — invisible in a photo feed. The lesson: time-based IDs trade strict monotonicity for no-coordination, and you must design downstream consumers to tolerate "roughly sorted," not "strictly sorted." Code that assumes IDs are a perfect clock will eventually be surprised.

### 5. The cross-shard query that hurt

Product wants "all photos liked by users this person follows, newest first." The follows span thousands of users, hence thousands of shards. There is no single place to run this query — the data is scattered by `user_id` across every machine. The team ends up scatter-gathering: query many shards in parallel, merge and sort the results in the application (the time-sortable IDs make the merge cheap — it's a k-way merge on the integer ID). The lesson, and the central limit of any sharding scheme: sharding optimizes single-shard access and penalizes cross-shard access. The ID's time-sortability softens the merge step, but it cannot make a fan-out query as cheap as a single-shard one. Choose the shard key so that your *most common* access pattern is single-shard.

### 6. The 1024-per-millisecond ceiling, examined and dismissed

During a capacity review, someone notices the 10-bit sequence caps a single shard at 1024 IDs per millisecond — about a million writes per second *per shard*. The worry: could a shard hit it? The math says no by a wide margin. With thousands of shards and a good key distribution, no single shard sees anything close to a million writes/sec; the aggregate write rate is spread across all of them. And even if a shard momentarily exceeded 1024 IDs in one millisecond, `next_id()` simply waits for the next millisecond tick — a sub-millisecond stall, not an error. The lesson: a bit-field design encodes its own limits explicitly, which is a feature — you can read the ceiling straight off the layout (`2^10` ids/ms/shard, `2^13` shards, `2^41` ms of epoch) and check it against your numbers, rather than discovering the limit in an outage.

### 7. The schema-count migration that didn't happen

A competitor's war story circulates: they sharded into 8 databases, outgrew them, and spent a quarter re-sharding into 32 while live. Instagram's equivalent quarter is a non-event, because the logical shard count was 8192 from the start. Growing from 4 machines to 40 is forty config entries and forty `pg_dump`s, run gradually, with zero changes to the sharding function or the ID format. The lesson, and the thesis of the whole design: **the expensive migration is the one you avoid by over-provisioning the cheap, immutable dimension (logical shards) on day one.** Logical shards cost almost nothing to create and freeze forever; physical machines are what you actually scale.

### 8. The connection storm at the pooler

As the machine count grows, a subtler problem surfaces: every application server wants a connection to every database it might route to, and Postgres connections are expensive (each is a backend process with its own memory). With dozens of app servers and a handful of database machines, the connection count balloons, and Postgres starts spending real CPU just context-switching between idle backends. The fix is a connection pooler — Instagram used PgBouncer — sitting between the app and Postgres, multiplexing thousands of client connections onto a small pool of real backend connections per machine in transaction-pooling mode. The self-routing IDs make this clean: the app still computes `(machine, schema)` by bit-shift, but it connects to the pooler for that machine rather than to Postgres directly. The lesson: sharding solves "too much data on one box," but packing many schemas onto few machines and many app servers in front of them creates a *connection* scaling problem that is separate, and a pooler is the standard answer. Routing-by-computation helps here too — there's no directory service adding its own connection fan-out.

### 9. The foreign key that crossed a shard

An engineer adds a `tagged_users` table relating a photo to the users tagged in it, and naively writes a foreign key from `tagged_users.user_id` to a `users` row. The problem: the photo lives in the *photo owner's* shard, but a tagged user lives in *their own* shard, on a different machine. A cross-machine foreign key is impossible — Postgres can only enforce constraints within one database. The resolution is to drop the database-level FK and enforce the relationship in the application, accepting that referential integrity across shards is your job, not the database's. The tagged user's ID still self-routes (bit-shift it to find their shard), so reads work fine; it's only the *constraint* that can't span shards. The lesson, true of every sharded system: sharding breaks the relational guarantees that hold within a single database. Joins, foreign keys, and transactions are single-shard luxuries; across shards you trade them for the application doing the bookkeeping. Design your tables so the relationships you must enforce stay inside one shard.

## When to reach for this design — and when not to

Reach for logical-shards-plus-self-routing-IDs when:

- **Your dominant access pattern is keyed by a natural entity** — user, tenant, account — so most queries hit a single shard. Instagram shards by `user_id` and most reads are "this user's photos."
- **You will grow the machine count over years** and cannot afford a re-shard each time. The large fixed logical shard count is insurance you buy once.
- **You control ID generation** and can make the primary key a 64-bit integer. Retrofitting this onto a schema that already issued random UUIDs as keys is a migration in itself.
- **You want routing to be a computation, not a service.** Bit-shift routing has no failure mode and no latency; it is the cheapest possible router.
- **You're on Postgres (or any engine with schemas and stored procedures)** and want the generator to live next to the data. The same idea works in other databases; the schema-as-shard trick is specifically Postgres-shaped. For the MySQL-with-Vitess flavor of the same problems, see [Vitess: sharding MySQL at scale](/blog/software-development/database/vitess-sharding-mysql-at-scale), and for how Notion and Figma applied schema-style sharding to their own Postgres, [Notion and Figma's Postgres sharding](/blog/software-development/database-scaling/notion-and-figma-sharding-postgres).

Skip it, or adapt it heavily, when:

- **Your queries are inherently cross-shard.** If the workload is analytics, graph traversal, or "find across all users," sharding by entity fights you, and the self-routing ID buys little. A different partitioning (or a separate analytics store) is the answer.
- **You have a hot key you can't avoid.** If one tenant or user dwarfs all others, no shard key fixes it, and the schema-granularity move from §5 can't split a single hot shard. You need application-level fan-out or a dedicated path for the whale.
- **You can't pick a stable shard key.** The shard key is frozen into every ID forever. If you genuinely don't know how you'll query the data, sharding this way prematurely is a bet you may lose. Build on one machine until the access pattern is clear.
- **Off-the-shelf already solves it.** Modern managed databases (Vitess, Citus, CockroachDB, Spanner) automate sharding and rebalancing. Instagram built this in an era when those weren't options at their scale; today, reach for a battle-tested system before hand-rolling bit fields — unless, as Instagram did, you have a reason the off-the-shelf option doesn't fit.

The enduring lesson has nothing to do with Postgres or with 13 bits. It is this: **make your unit of partitioning small and numerous and immutable, make your unit of capacity separate and movable, and make your routing key carry its own address.** Do that, and growth stops being a migration. Instagram proved you could buy a decade of scaling headroom with three integer fields and a stored procedure — and that the cheapest re-shard is the one your day-one design made unnecessary.
