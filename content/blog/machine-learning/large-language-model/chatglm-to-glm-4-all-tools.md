The first two articles in this series were about a base model: [the blank-infilling objective](/blog/machine-learning/large-language-model/autoregressive-blank-infilling-glm) that made GLM, and [the engineering](/blog/machine-learning/mlops/engineering-glm-130b) that scaled it to 130B without diverging. But a base model is not a product. The distance between "a 130B model that completes text" and "an assistant that browses the web, runs Python, and resolves your request autonomously" is a second, entirely different body of engineering — and it is the subject of this article.

This is the third stage of the [GLM lineage](/blog/machine-learning/large-language-model/glm-lineage-frontier-llm-technique): the eighteen months from ChatGLM-6B (March 2023) to GLM-4 All Tools (2024), documented mainly in the [ChatGLM technical report](https://arxiv.org/abs/2406.12793). Where the GLM-130B story was about *not diverging*, this story is about *alignment* — turning a capable base model into something useful, safe, and agentic. The failure modes are completely different. A divergent loss curve kills a base model; a poorly-aligned model ships and then disappoints in a thousand small ways: it refuses the wrong things, it hallucinates tool calls, it loses the thread on turn four, it can't use the 128K context you paid for.

The GLM team's answers — authentic-human SFT data over synthetic, a productionized RLHF system with three specific stabilizers, a long-context alignment recipe that fixes a subtle loss-weighting bug, an agent-tuning trick that adds skill without erasing generality, and a model that learns to choose its own tools — are each independently stealable, and each is the kind of hard-won detail that only shows up in a report written by people who actually shipped the thing. Let's walk them, one quiet failure mode at a time.

## Why "alignment is just SFT then RLHF" undersells it

The compressed version of post-training — "do supervised finetuning, then RLHF, ship it" — is technically accurate and almost useless, because all the difficulty is in the details that compression hides. Here is the mismatch:

| Assumption | The naive view | The reality in the GLM-4 report |
| --- | --- | --- |
| "SFT data — just generate a lot of it" | Synthetic instructions scale cheaply | *Authentic human* prompts beat template/model-generated ones for alignment quality |
| "RLHF is one algorithm" | Run PPO and you're done | PPO at scale needs reward-variance reduction, fused-gradient parallelism, and forgetting regularization or it collapses |
| "Long context is a positional trick" | Extend RoPE and you're done | Naive packing silently *mis-weights the loss*; LongAlign fixes the weighting |
| "Agent skill — finetune on agent data" | Train on trajectories | Pure agent data erases general ability; the 1:4 mixing ratio is the actual technique |
| "Tool use is function-calling syntax" | Define a JSON schema | All Tools is *autonomous selection* — the model decides which tool, trained to plan |

Every row is a section. The through-line is that each "obvious" step has a non-obvious failure mode, and the GLM team documented both.

This is, in a sense, the defining difference between the *base-model* engineering of the [previous article](/blog/machine-learning/mlops/engineering-glm-130b) and the *alignment* engineering here. Base-model failures are loud — the loss diverges, the run crashes, you know immediately. Alignment failures are quiet — the model ships and is subtly disappointing, and you might not pin down *why* for weeks. A model that's been badly aligned doesn't crash; it just refuses reasonable requests, hallucinates confidently, loses coherence on long conversations, and routes to the wrong tool. None of those throw an exception. So alignment engineering is, more than base-model engineering, about anticipating failure modes you won't be able to see in a loss curve, and building the evaluation (often custom benchmarks) to surface them. The GLM-4 report is valuable precisely because it names the quiet failures and their fixes, converting "alignment is finicky" into a concrete list of specific problems and specific solutions.

## The mental model: a staircase of capabilities

![A timeline of the ChatGLM family from ChatGLM-6B through GLM-4-9B, each release adding one capability](/imgs/blogs/chatglm-to-glm-4-all-tools-1.png)

The timeline above is the mental model: the family is a *staircase*, each step adding one capability while keeping the last. ChatGLM-6B (March 2023) was the first aligned chat model. ChatGLM2 (June 2023) added multi-query attention and 32K context. ChatGLM3 (October 2023) added the function-calling prompt format. GLM-4 (2024) added grouped-query attention, 128K context, and All Tools. GLM-4-9B (June 2024) opened the weights with up to 1M context and 26 languages. No step throws away the previous one — exactly the accretion pattern the [survey](/blog/machine-learning/large-language-model/glm-lineage-frontier-llm-technique) identified across the whole lineage. The compression of eighteen months into five releases is itself a discipline: each release shipped one well-tested capability rather than a grab-bag, which kept the engineering attributable and the regressions rare. A team that tries to add context, attention efficiency, tools, and alignment all in one release ships a confounded mess; shipping them one at a time is how you keep knowing which change caused which effect.

![A matrix showing which capability each ChatGLM version added across context, attention, tools, and alignment](/imgs/blogs/chatglm-to-glm-4-all-tools-9.png)

The matrix above is the same story sliced the other way: read *down* a column and you watch one capability mature across the family. Context goes 2K → 32K → 32K → 128K-1M. Attention goes MHA → MQA → MQA → GQA. Tools go none → none → function-call → All Tools. Alignment goes SFT+RLHF → improved → agentic RLHF. The diagonal march from top-left (gray, basic) to bottom-right (green, mature) is the eighteen months in one picture. The rest of this article is a tour of the columns.

One clarification on naming, because the GLM-4 generation has several variants that the report and the leaderboards refer to interchangeably. **GLM-4** is the flagship, and it shipped in dated revisions — GLM-4 (0116) was the January 2024 version, GLM-4 (0520) the stronger May version that most "GLM-4" benchmark numbers refer to. **GLM-4-Air** (0605) is the cost-efficient tier — a smaller, cheaper flagship variant for latency- and price-sensitive deployments. **GLM-4-9B** is the open-weight model (June 2024), released with `-Chat`, `-Chat-1M`, and the multimodal `GLM-4V-9B` companion, all sharing the flagship's post-training pipeline. The report notably does *not* disclose the flagship's parameter count, layer count, or exact token mix — only "~10 trillion tokens" and the 150K vocabulary — so any specific flagship parameter number you see quoted is speculation. When this article cites a GLM-4 benchmark, it's GLM-4 (0520) unless noted; when it discusses architecture specifics, those are most reliably grounded in the open GLM-4-9B, whose card discloses 151,552 vocabulary tokens and 26 languages.

## 1. Architecture: shrinking the KV cache to buy context

> **Senior rule of thumb:** long context is an *attention-economics* problem before it is a positional-encoding problem. Every token you want to attend to costs KV-cache memory, and the KV cache is what runs out first.

The architecture changes from ChatGLM-6B to GLM-4 are mostly in service of one goal: make the KV cache small enough that long context fits.

![A before-after comparing multi-head attention in ChatGLM-6B with grouped-query attention in GLM-4](/imgs/blogs/chatglm-to-glm-4-all-tools-2.png)

The before-after above shows the arc. ChatGLM-6B used standard multi-head attention (MHA): every one of its 96 attention heads has its own key and value projections, so the KV cache stores 96 keys and 96 values per token. That cache is what you must keep in GPU memory for every token in the context, and at 96 heads it grows fast — which is why ChatGLM-6B topped out at 2K context.

ChatGLM2 took the aggressive step to **multi-query attention (MQA)**: all query heads share a *single* key-value head. The KV cache shrinks by ~96×, which is what let context jump to 32K, and inference sped up ~42%. But MQA costs some quality — collapsing to one KV head is a real information bottleneck. So GLM-4 settled on the middle ground that the whole field converged on: **grouped-query attention (GQA)** with 8 KV groups. The 96 query heads share 8 key-value heads (12 queries per group), which keeps the cache ~12× smaller than MHA while recovering nearly all of MHA's quality. That's the combination that made 128K — and the experimental 1M — context affordable.

The rest of the GLM-4 architecture is the modern-transformer consensus, with a few choices worth lifting:

| Choice | Value | Why it matters |
| --- | --- | --- |
| Attention | GQA, 8 KV groups | small KV cache for long context |
| Positional encoding | 2D-RoPE | rotary, extended to GLM's 2D scheme |
| Normalization | RMSNorm | cheaper than LayerNorm, no mean subtraction |
| FFN activation | SwiGLU | gated; needs the FFN-dim trick below |
| FFN dimension | 10/3 × hidden | sized so SwiGLU keeps param count constant |
| Vocabulary | 150,000 | byte-level BPE merged with `cl100k_base` |
| Bias terms | none except QKV | fewer params, marginally more stable |

Two are worth dwelling on. The **vocabulary** merges GLM's own byte-level Chinese/multilingual BPE with OpenAI's `cl100k_base` (the tiktoken vocabulary) — a pragmatic way to inherit strong English tokenization rather than re-deriving it, which matters for a bilingual model that must be competitive in both languages. And the **FFN sized at 10/3 × hidden** is the bookkeeping that makes SwiGLU free: SwiGLU adds a third projection compared to a standard FFN, so to keep the parameter count constant you shrink the FFN dimension from the usual 4× to 10/3× (≈ 3.33×). These are the unglamorous details that never make the abstract but decide whether your model fits in memory.

### The KV-cache arithmetic that makes context expensive

It's worth doing the arithmetic that makes the GQA decision so consequential, because "the KV cache is what runs out first" is a claim you should be able to derive, not just assert. The KV cache stores, for every token in the context, a key vector and a value vector *per KV head*, in every layer. Its size is roughly `2 × n_layers × n_kv_heads × head_dim × seq_len × bytes_per_element`. The two parts that move are `n_kv_heads` and `seq_len`, and they multiply.

For a concrete feel, take a GLM-4-class model with ~40 layers and a head dimension of 128, in FP16 (2 bytes). With full MHA at 96 KV heads, one token of context costs `2 × 40 × 96 × 128 × 2 ≈ 2.5 MB`. At 128K context that's `2.5 MB × 128K ≈ 320 GB` — for a *single* request's KV cache, which is absurd; it dwarfs the model weights. With GQA at 8 KV heads, the same token costs `2 × 40 × 8 × 128 × 2 ≈ 210 KB`, and 128K context is `~27 GB` — large but tractable, and shrinkable further with KV-cache quantization. That 12× reduction (96 → 8 KV heads) is *exactly* the ratio between "128K context is impossible" and "128K context fits." This is why the attention-head architecture and the context window are not independent choices: the KV-head count *sets* the maximum affordable context. GLM's MHA → MQA → GQA arc is the team discovering, release by release, where on that tradeoff curve to sit. MQA (1 KV head) is the most aggressive — smallest cache, biggest quality hit; GQA at 8 is the sweet spot the whole field converged on. For the deep version of this, the blog's [KV-cache](/blog/machine-learning/large-language-model/kv-cache) and [KV-cache optimization](/blog/machine-learning/large-language-model/kv-cache-optimization-and-management) deep-dives are the companions.

### Why GQA recovers MHA's quality

The natural question is *why* GQA loses so little quality despite throwing away most of its KV heads. The intuition: the query heads are where most of the model's representational diversity lives — different query heads attend to different things (syntax, coreference, position). The key-value heads are more redundant; many of them encode similar information. MQA's mistake is collapsing all that redundancy into a *single* KV head, which loses the bit of genuine diversity that does exist among KV heads. GQA keeps 8 KV heads — enough to preserve the real diversity — while sharing each across 12 query heads, which is where the redundancy was anyway. So GQA is, in effect, a *compression* of the KV heads that keeps the signal and drops the redundancy, rather than MQA's lossy collapse. The empirical result across the field — GQA at a handful of groups matches MHA within noise — is the validation that KV-head redundancy is real and exploitable. For GLM-4 specifically, it's the architectural enabler of the entire long-context story; without it, the 128K and 1M context windows that define the GLM-4-9B release would have been economically impossible. For how GLM-4's choices compare to Qwen, Llama, and DeepSeek, see [modern LLM architectures](/blog/machine-learning/large-language-model/modern-llm-architectures-qwen-llama-gemma-deepseek), and for the tokenizer rabbit hole, [designing and choosing a tokenizer](/blog/machine-learning/large-language-model/designing-choosing-tokenizer-llm).

### RMSNorm and SwiGLU: the boring choices that compound

The normalization and activation choices look minor on the spec sheet but are worth understanding because they're the modern consensus for good reasons. **RMSNorm** replaces LayerNorm's two operations (subtract the mean, divide by the standard deviation) with just one (divide by the root-mean-square). Dropping the mean-subtraction turns out to cost almost nothing in quality while saving a reduction operation per normalization — and a transformer normalizes twice per layer, so across 40+ layers that adds up to a real throughput win. It's the kind of optimization that's individually trivial and collectively meaningful, which is why it became standard. **SwiGLU** is a gated activation: instead of the FFN being `W2 · GeLU(W1 · x)`, it's `W2 · (Swish(W1 · x) ⊙ (W3 · x))` — a third projection `W3` provides a multiplicative gate. Gating gives the FFN a cheap form of input-dependent selectivity that consistently improves quality. The catch is that third projection: it adds parameters, which is why GLM-4 sizes the FFN at 10/3 × hidden instead of the usual 4× — shrinking the hidden FFN dimension exactly enough that adding `W3` leaves the total parameter count unchanged. This is a recurring theme in well-engineered models: an architectural upgrade is only "free" if you re-budget the parameters it adds, and the teams that do this carefully ship models that are strictly better at the same size rather than slightly bigger and marginally better. The discipline of holding parameter count constant while improving the architecture is what lets you attribute gains to the *idea* rather than to scale.

## 2. Alignment: the SFT data stance and a productionized RLHF

> **Senior rule of thumb:** the base model sets the ceiling; alignment decides how much of that ceiling you actually deliver. And the highest-leverage alignment decision is usually about *data*, not algorithm.

![A pipeline of the GLM-4 alignment stages: base model, SFT, reward model, RLHF, All Tools alignment, aligned assistant](/imgs/blogs/chatglm-to-glm-4-all-tools-3.png)

The pipeline above is the alignment journey: a ~10T-token base model, supervised finetuning on human data, a reward model trained from preferences, RLHF that uses it, and a final alignment pass for autonomous tool use. Each stage fixes something the previous one couldn't. Let's take the two that carry the most reusable technique.

There's a structural point worth making before the details: notice that the pipeline is *staged*, and each stage's output is the next stage's input. SFT produces a model good enough to generate reasonable responses, which is a prerequisite for collecting useful preference pairs (you can't rank responses if they're all garbage). The reward model trained on those pairs is a prerequisite for RLHF. And a competent RLHF'd chat model is a prerequisite for the All Tools alignment, because you can't teach autonomous tool use to a model that can't follow instructions. This dependency chain is why post-training is a *pipeline* and not a grab-bag — you can't reorder the stages, and weakness in an early stage caps everything downstream. A common mistake is to under-invest in SFT (rushing to the "interesting" RL) and then wonder why RLHF won't converge; the answer is usually that the SFT model wasn't good enough to bootstrap clean preferences.

### The SFT data stance: authentic humans over synthetic templates

The GLM-4 report contains one sentence that is worth more than most alignment papers, because it is a falsifiable design stance stated plainly:

> In SFT, we find that authentic human prompts and interactions, instead of template-based or model-generated responses, are vital to the alignment quality.

Sit with how contrarian this was. In 2024, the field was awash in synthetic instruction data — generate a million instructions with a strong model, finetune on them, scale cheaply. The GLM team's measured position was that *real* human prompts and interactions carry an alignment signal that templates and model-generated responses do not. The intuition is that synthetic data inherits the generating model's blind spots and stylistic tics; it teaches your model to imitate another model rather than to satisfy a human. Authentic human interactions, by contrast, contain the messy, surprising, out-of-distribution requests real users actually make — and the alignment that matters is alignment to *those*.

This is a cheap technique to steal, because it's a discipline rather than a trick: before you generate ten million synthetic SFT examples, ask whether the marginal *authentic* example is worth more than the marginal synthetic one. Often it is, and the right move is to spend on data collection and annotation rather than on a bigger generator. The responses themselves were scored on a rubric — safety, factuality, relevance, helpfulness, human preference — which is the other half of the discipline: define what "good" means before you label.

It's worth unpacking why synthetic data has the failure mode the GLM team is guarding against, because the mechanism is subtle. When you generate SFT data with a strong model, every example is filtered through that model's distribution: its phrasings, its hedges, its refusal patterns, its blind spots. Finetune on a million such examples and your model learns to *imitate the generator* — which is fine if the generator is what you want to be, but it caps you at the generator's ceiling and inherits all of its quirks. Worse, the prompts in synthetic data tend to be *clean* — well-formed, unambiguous, the kind of question a model would write — whereas real human prompts are messy: typo-ridden, underspecified, weirdly phrased, full of implicit context. Alignment that matters is alignment to the messy real prompts, and you only get those from real humans. This is also why the report pairs the data stance with a *rubric*: authentic prompts give you the right *distribution* of requests, and the rubric ensures the *responses* you train on are good across the dimensions you care about. Distribution from humans, quality from the rubric. The five rubric dimensions — safety, factuality, relevance, helpfulness, human preference — are themselves worth noting because they're nearly orthogonal: a response can be safe but unhelpful, helpful but inaccurate, accurate but irrelevant to what was asked. Scoring each dimension separately (rather than a single "is this good?" score) gives the training signal more resolution and lets you diagnose *which* axis a model is weak on. A model that scores high on helpfulness and low on factuality has a different problem (it's confidently wrong) than one high on factuality and low on relevance (it answers a different question than asked), and the fixes differ. Multi-dimensional rubrics are more annotation work, but they turn "the model is bad" into "the model is unsafe on these prompts and irrelevant on those," which is actionable.

There's a complementary technique the GLM lineage uses that's worth naming: **Self-Contrast**, a feedback-free alignment method where the target model self-generates large numbers of *negative* samples to build preference pairs without human or reward-model labels. The idea is that a model is often better at *recognizing* a bad response than at avoiding generating one, so you can have it produce contrastive negatives cheaply and learn from the contrast. It's part of the same philosophy: get the highest-quality signal you can at the lowest labeling cost, whether that means investing in authentic human data where it matters or bootstrapping preference pairs from the model itself where human labels are too expensive. The art of alignment-at-scale is largely the art of allocating a finite labeling budget across these options: authentic human data where the distribution and texture matter most, synthetic data where you need verifiable capability cheaply, and self-generated contrastive data where human labels are too expensive but the model can judge. A team that treats "get more data" as a single undifferentiated goal will over-spend on whichever source is easiest (usually synthetic) and under-spend on the authentic data that actually moves alignment. The teams that ship the best-aligned models are the ones that think of labeling budget the way a portfolio manager thinks of capital — allocated deliberately across sources with different risk and return, not poured into the cheapest option.

### ChatGLM-RLHF: making PPO survive production

SFT teaches the model the *format* of good responses; it cannot teach the things that require comparing responses against each other. That's RLHF's job, and the GLM team built a real one, documented in the companion paper [ChatGLM-RLHF](https://arxiv.org/abs/2404.00934). What RLHF fixes that SFT can't: response rejection (declining bad requests), safety, the bilingual token-mixing problem (the model randomly switching languages mid-response), and multi-turn coherence.

![A graph of the ChatGLM-RLHF system, routing the reward through three stabilizers before policy optimization](/imgs/blogs/chatglm-to-glm-4-all-tools-4.png)

The graph above is the system. Preference pairs train a reward model; the reward signal then passes through **three stabilizers** before policy optimization, and those stabilizers are the actual engineering contribution, because PPO at this scale is notoriously prone to collapse:

- **Reward-variance reduction.** PPO's gradient variance is dominated by the variance of the reward signal. High-variance rewards make the policy lurch unpredictably. Reducing reward variance (through normalization and careful reward shaping) keeps the updates smooth, which is often the difference between a run that converges and one that oscillates into nonsense.
- **Fused-gradient model parallelism.** RLHF is memory-brutal because you hold multiple models in memory at once — the policy, the reference, the reward model, and the value model. Fusing the gradient computation across the model-parallel shards keeps this tractable. This is the systems half of RLHF, the part that doesn't make the algorithm diagrams but determines whether the algorithm runs at all.
- **Forgetting regularization.** The classic RLHF failure: the policy optimizes the reward so hard that it forgets the general capabilities the SFT model had — it becomes a reward-hacking specialist that aces the preference model and fails everything else. A regularization constraint that penalizes drifting too far from the SFT model keeps the general capabilities intact. (The KL-to-reference penalty in standard PPO is the basic form of this; GLM-4's framing makes "don't forget" an explicit goal.)

The payoff: ChatGLM-RLHF reports a **~15% average win-rate gain over the SFT model** on Chinese alignment tasks.

### Why RLHF is so hard to run, mechanically

To appreciate the stabilizers you have to appreciate why RLHF is mechanically brutal in a way SFT is not. In SFT you hold *one* model in memory and do supervised gradient steps — boring and stable. In PPO-based RLHF you hold *four* models simultaneously: the **policy** (the model you're training), a frozen **reference** policy (to compute the KL penalty that keeps you from drifting too far), the **reward model** (to score generations), and a **value model** (to estimate the advantage). For a large model, four copies is a memory catastrophe, which is what the *fused-gradient model parallelism* addresses — it makes holding and updating these models across model-parallel shards tractable. On top of the memory problem, the training loop is *on-policy*: you generate rollouts from the current policy, score them, and update — and the distribution you're learning from shifts every step as the policy changes. That non-stationarity is what makes PPO prone to oscillation and collapse, and it's what *reward-variance reduction* tames: a noisy reward on a shifting distribution is a recipe for the policy lurching in the wrong direction and never recovering.

The third stabilizer, *forgetting regularization*, deserves its own paragraph because it's the most conceptually important. The whole danger of RLHF is that the reward model is a *proxy* for what you want, and any proxy can be gamed. Optimize the reward hard enough and the policy finds the cracks — it discovers responses that score high on the reward model but are actually worse (overly long, sycophantic, formulaic), and in chasing them it drifts away from the general competence the SFT model had. This is reward hacking, and it manifests as a model that aces your preference benchmark while quietly getting worse at everything you didn't measure. The standard defense is the KL-to-reference penalty (stay close to the SFT model); GLM-4's framing of "forgetting regularization" makes the goal explicit — the constraint isn't just regularization for stability, it's specifically there to *preserve the capabilities you already have*. The practical signal to watch is a divergence between your reward-model score (going up) and a held-out general benchmark (going down); when you see that gap open, your forgetting regularization is too weak. This reward-vs-general-benchmark gap is, in my experience, the single most important dashboard in an RLHF run — more important than the reward curve itself, because the reward going up is *expected* and tells you little, while the general benchmark holding steady (or not) is what tells you whether the reward gains are real or hacked.

For the conceptual grounding on PPO vs DPO vs GRPO — note that GLM-4 uses *both* PPO and DPO — the [GRPO vs DPO vs PPO decision guide](/blog/machine-learning/large-language-model/grpo-vs-dpo-vs-ppo-decision-guide) is the prerequisite, and the [DPO finetuning walkthrough](/blog/machine-learning/large-language-model/fine-tuning-llm-with-dpo) covers the direct-preference path GLM-4 also used. The RL infrastructure required to do this at frontier scale is the subject of a later article in this series, on GLM-4.5's slime stack; ChatGLM-RLHF is the smaller, earlier ancestor of that system.

## 3. Long context: the LongAlign recipe

> **Senior rule of thumb:** extending the context window is two problems — can attention reach that far (positional), and does the model *use* the long context to follow instructions (alignment). The second is the hard one, and it hides a subtle loss bug.

Getting a model to 128K is not just a matter of training on long documents. GLM solved the alignment half with [LongAlign](https://arxiv.org/abs/2401.18058), and the techniques are sharper than the usual "train on long sequences" hand-wave.

![A before-after of LongAlign: naive long-context tuning versus packing with sorted batching and loss re-weighting](/imgs/blogs/chatglm-to-glm-4-all-tools-6.png)

The before-after above shows the problem and the fix. Long instruction data is wildly variable in length — some examples are 2K tokens, some are 100K — and that variability creates two problems that LongAlign solves together:

- **Packing with sorted batching.** If you pad every sequence to the maximum length, you waste enormous compute on padding tokens. LongAlign packs multiple sequences into one batch and sorts by length to minimize padding. That's a throughput win that's easy to underestimate: without it, a batch containing one 100K sequence forces every other sequence in the batch to pad to 100K.
- **The loss-weighting fix** — the subtle one. When you pack *k* sequences into one batch and compute a normal mean loss over all tokens, a packed batch containing one long sequence and several short ones *under-weights* the short sequences relative to an unpacked run. The long sequence's many tokens dominate the mean, so the model effectively learns more from long examples and less from short ones than you intended. LongAlign introduces a loss weighting that **restores each sequence's correct contribution under packing** — each sequence counts the same regardless of how it was packed.

This bug is worth internalizing because it generalizes: *any time you change the batching for throughput, re-derive what your loss is actually averaging over.* Efficiency hacks love to smuggle in a silent reweighting, and a silent reweighting is a silent change to your training objective. LongAlign reports up to **+30% on long tasks** while preserving short-task quality, and it shipped **LongBench-Chat** (a 10K–100K-token evaluation) because — as we'll see in the data section — the capability didn't have a good public ruler. The open GLM-4-9B-Chat-1M pushes this to a 1M-token context, best read as an existence proof than a production guarantee, but the recipe that reaches it is the reusable artifact.

### The loss-weighting fix, derived

The loss-weighting bug is worth making fully concrete, because it's the kind of thing you'll only catch if you've seen it once. Suppose you pack two sequences into one batch: sequence A has 10 supervised tokens, sequence B has 1,000. If you compute the standard mean cross-entropy over all 1,010 tokens, sequence B contributes 1,000/1,010 ≈ 99% of the loss and sequence A contributes ~1%. Now compare to the *unpacked* world, where A and B are separate batches: each contributes 100% of its own batch's loss, and if you weight batches equally, A and B contribute *equally* to the overall gradient. So packing has silently changed A's effective weight from 50% (equal) to ~1% (proportional to token count). Your long sequences now dominate training and your short ones are nearly ignored — a real change to your objective that no error message will report.

LongAlign's fix is to re-weight the per-token loss so that each *sequence* contributes the amount it would have in the unpacked, equal-batch world — roughly, scale each sequence's token losses by a factor inversely proportional to its length (and proportional to the number of sequences in the pack), so a 10-token sequence and a 1,000-token sequence end up with the same total weight. The exact weighting depends on what you want "equal" to mean (equal per-sequence, or equal per-token across the dataset), and that choice is itself a modeling decision — but the *default* mean-over-packed-tokens is almost never what you want once sequence lengths vary by orders of magnitude. The general principle, stated once more because it's that important: **packing changes your loss's implicit weighting, so you must re-derive and restore the weighting you actually intended.** This applies to any throughput optimization that changes how examples are grouped — packing, dynamic batching, sequence bucketing — and it's one of the most common silent bugs in efficient training.

The positional half of long-context extension — getting attention to *reach* 128K — is the more familiar part: extend the RoPE base frequency and continue pretraining on long documents so the model learns to use the extended positions. GLM-4 does this, but the report's emphasis is squarely on the alignment half, because that's where the non-obvious work is. Anyone can extend RoPE; the LongAlign loss-weighting is the part that makes the long context actually *useful* for following instructions.

There's a sharp distinction hiding here that's worth making explicit: a model can have a long context window and still be bad at *using* it. The window is the positional capacity — how far attention can reach. The skill is whether the model actually attends to the right part of a long document when answering. You can extend a window to 128K and find the model still anchors on the first and last few thousand tokens, ignoring the middle (the well-documented "lost in the middle" pathology). LongAlign targets the *skill*: by training on long instruction data where the relevant information is genuinely distributed across the long context — and weighting the loss correctly so the model learns from those examples — it teaches the model to actually use the window it has. This is why the long-context story is an *alignment* story, not just an architecture story. The architecture gives you the window; the alignment data and loss weighting give you the skill to use it. Evaluating only the window (can it ingest 128K without crashing) while ignoring the skill (does it answer correctly from token 60,000) is how teams ship long-context models that benchmark well on retrieval and disappoint on real long documents.

## 4. Agentic: All Tools and the AgentTuning trick

> **Senior rule of thumb:** adding a narrow skill (tool use) to a generalist is dangerous, because the narrow skill tends to cannibalize the broad one. The technique is in the *mixing*, not the data.

This is where GLM-4 becomes more than a chatbot. "All Tools" is the capability that lets it browse, compute, and act — and it was assembled over three years.

![A graph of GLM-4 All Tools autonomously routing a request to a browser, Python interpreter, image generation, or user functions, then answering](/imgs/blogs/chatglm-to-glm-4-all-tools-5.png)

The graph above is the All Tools mental model. The crucial word is **autonomous**: GLM-4 All Tools *decides on its own* which tool a request needs. It is not a fixed pipeline where you tell it "use the browser now"; it plans, routes the request to the web browser, Python interpreter, image generator (CogView), or a user-defined function/API, reads the result, and continues — possibly chaining multiple tools — until it can answer. The report describes it as a model that "autonomously decides when and which tool(s) to use." That autonomy is the difference between a tool-augmented chatbot and an agent.

To see why autonomy is hard, contrast it with plain function-calling. In a function-calling setup, the *developer's harness* owns the control flow: the application decides "this is a weather query, call the weather API," and the model just fills in the arguments. The model never decides *whether* to use a tool — that's the developer's job. Autonomous tool use moves that decision *into the model*: given an arbitrary request, the model must recognize "this needs current information, I should browse" or "this is a calculation, I should run Python" or "this is just a chat, I should answer directly." That recognition is a *planning* skill, and it's qualitatively harder than argument-filling because it requires the model to reason about its own capabilities and limitations. A model that browses when it should have just answered is annoying; one that answers from memory when it should have browsed hallucinates. Getting the routing right — and knowing when *no* tool is needed — is most of the difficulty.

The multi-turn dimension makes it harder still. Real agentic tasks aren't one tool call; they're a *loop*: browse, read the result, realize you need a different search, browse again, then compute on what you found. The model has to maintain a plan across turns, incorporate tool outputs into its reasoning, and decide when it has enough to answer. This is why the report frames All Tools as planning "complex instructions and chaining one or multiple tools" — the chaining is the agentic part, and it's where most agent systems break in practice. Each turn the model must parse the previous tool's output (which might be an error, a partial result, or an unexpected format), update its plan, and decide the next action — and errors compound, because a wrong turn three feeds bad context into turns four through ten. This is why agentic reliability is so much harder than single-turn accuracy: a model that's 95% reliable per turn is only `0.95^10 ≈ 60%` reliable over a ten-turn task, so small per-turn error rates become large task-failure rates. Training for multi-turn robustness — recovering from a bad tool result, not getting stuck in a loop, knowing when to give up — is a distinct competence that the agentic RL of [GLM-4.5](/blog/machine-learning/large-language-model/glm-lineage-frontier-llm-technique) would later target directly with verifiable rewards over whole trajectories. The serialization format matters here too: when a tool's arguments are *code* (as they often are for the Python interpreter), a naive JSON encoding forces the model to escape quotes and backslashes, which it frequently gets wrong. This is exactly the problem GLM-4.5 later solves with an XML-style tool-call template; the seeds of that fix trace back to the function-call format work in this generation. For the broader agentic-framework landscape, the blog's [LangGraph vs ReAct comparison](/blog/machine-learning/ai-agent/langgraph-vs-react-vs-agentic-frameworks) and [fine-tuning tool-calling LLMs](/blog/machine-learning/training-techniques/fine-tuning-tool-calling-llms-when-how) are the companions.

How do you train autonomous tool selection without breaking the model's general ability? The keystone is [AgentTuning](https://arxiv.org/abs/2310.12823).

![A tree of AgentTuning: the AgentInstruct dataset of 1866 trajectories across 6 environments, mixed 1:4 with general data, producing AgentLM](/imgs/blogs/chatglm-to-glm-4-all-tools-7.png)

The tree above decomposes AgentTuning, and the single most transferable number in it is the **1:4 mixing ratio**. The method:

- **AgentInstruct**, a dataset of **1,866 GPT-4-generated, reward-filtered agent trajectories** across six environments (ALFWorld, WebShop, Mind2Web, knowledge graph, OS, database). Note how *small* this is — under two thousand trajectories. Agent skill doesn't require a million examples; it requires a few high-quality ones.
- **The 1:4 ratio.** The agent trajectories are mixed with general ShareGPT-style instructions at roughly one part agent to four parts general. This is the actual technique. Train on agent data *alone* and you get a model that's a competent agent and has forgotten how to chat — the narrow skill cannibalizes the broad one. The 1:4 dilution is what lets AgentLM-70B reach GPT-3.5-turbo's agent performance on *unseen* environments **without** regressing on general benchmarks.

The lesson generalizes far beyond agents: whenever you graft a narrow capability onto a generalist, the mixing ratio with general data is a first-class hyperparameter, and the right answer is usually "less narrow data than you'd think." The same agent groundwork also includes [WebGLM](https://arxiv.org/abs/2306.07906) (retrieval-augmented browsing, the precursor to All Tools' web capability) and the ChatGLM3 function-calling format — three years of agentic pieces that cashed out at once in GLM-4 All Tools. AgentTuning is also a clean example of how to *bootstrap* a capability you don't yet have: the trajectories were generated by GPT-4 (a model that already had decent agency) and then reward-filtered, so a weaker model could learn agency by imitating a stronger one's successful trajectories. This teacher-student bootstrapping — use the best available model to generate high-quality demonstrations of a skill, filter for success, then train a smaller or open model on them — is a pattern that recurs across the whole field, and it's how capability propagates from frontier models to the open ecosystem. The reward-filtering is the non-obvious half: GPT-4's trajectories aren't all good, so filtering for the ones that actually succeeded is what turns "imitate GPT-4" into "imitate GPT-4's *good* behavior."

### Why the environment diversity matters more than the count

The six environments in AgentInstruct — ALFWorld (embodied household tasks), WebShop (online shopping), Mind2Web (web navigation), a knowledge-graph environment, an operating-system environment, and a database environment — are chosen for *diversity*, not coverage of any single domain. This is the deeper design principle behind the small dataset size. Agent skill is fundamentally a *transferable* competence: planning, reading feedback, recovering from errors, deciding when you're done. You don't teach those by drilling a thousand examples of one environment; you teach them by showing the model the same *pattern* — observe, plan, act, observe — across environments that are superficially completely different. The model learns that the pattern generalizes precisely *because* the environments don't share surface features. An agent trained only on WebShop would learn "shopping," but an agent trained on six diverse environments learns "being an agent," which is what transfers to the unseen environments in the eval.

This is why the **reward-filtering** of the trajectories matters so much: with only 1,866 examples, each one carries a lot of weight, so a bad trajectory (one where GPT-4 made a planning error or took a clumsy path) teaches a bad pattern that the model will over-fit. Filtering trajectories by reward — keeping only the ones that actually succeeded efficiently — ensures the small dataset teaches *good* agency rather than mediocre agency. The combination is the recipe: few examples, maximally diverse environments, ruthlessly quality-filtered. The result, AgentLM (built on Llama-2 at 7B/13B/70B), reaches GPT-3.5-turbo-level agent performance on held-out tasks while preserving general ability — which is the existence proof that the recipe works, and the template GLM-4 All Tools scaled up. The blog's [agent curriculum and expert iteration](/blog/paper-reading/ai-interpretability/automatic-curriculum-expert-iteration-for-reliable-llm-reasoning) note explores the related idea of generating and filtering trajectories for reliability.

## 5. Results: matching GPT-4 where it counts

The alignment only matters if the aligned model is good, and on the 2024 landscape GLM-4 was genuinely competitive with GPT-4.

![A matrix of GLM-4 versus GPT-4 Turbo across MMLU, GSM8K, AlignBench, function calling, and long context](/imgs/blogs/chatglm-to-glm-4-all-tools-8.png)

The matrix above is the scorecard. On the hardest pure-reasoning benchmarks, GLM-4 (0520) trails GPT-4 Turbo slightly — MMLU 83.3 vs 86.7, GSM8K 93.3 vs 95.6. But on the axes that this article is about — alignment and tool use — it *ties*: **AlignBench 8.00, exactly matching GPT-4 Turbo**; **BFCL function calling 81.76 vs 81.24** (GLM-4 ahead); **LongBench-Chat 87.3 vs 87.2** (a tie on long context). And on **AgentBench overall (3.79)**, GLM-4 was the best model in the table, ahead of GPT-4 Turbo's 3.77.

The single most striking result is on autonomous browsing. On information-seeking tasks using the browser tool, **GLM-4 All Tools scored 78.08 versus GPT-4's 67.12** — a clear win on a capability the GLM team had been building toward since WebGLM in 2023. When you've shipped retrieval-augmented browsing, agent tuning, and a function-call format over three years, you've accumulated exactly the pieces needed for autonomous tool use, and the All-Tools browsing win is that groundwork cashing out. The All Tools Python-interpreter results tell the same story: GSM8K with a Python interpreter hits 91.59 (vs GPT-4's 92.72) and Math23K 88.50 (vs 88.40) — essentially parity on tool-augmented math.

It's also worth noting *which* benchmarks the GLM team chose to highlight, because benchmark selection is itself a statement of values. The report leans on AlignBench (alignment), BFCL (function calling), LongBench-Chat (long context), and AgentBench (agency) — the axes a *deployed assistant* is judged on by its users — rather than over-indexing on the contest-math and trivia benchmarks that dominate leaderboard marketing. That's a coherent stance: GLM-4 was built to be *used*, and the evaluation reflects that. When you read a model report, the benchmarks the team chose to foreground tell you what they optimized for, and a report that foregrounds alignment and tool use over raw trivia is telling you it's a product, not a leaderboard entry. The contrast with the next generation is instructive — GLM-4.5 would foreground SWE-bench and TAU-bench, signaling a shift toward *agentic and coding* as the capabilities that matter, which is exactly the frontier the field moved to.

The honest reading is that GLM-4 is a model that chose *where* to compete. It conceded a little on raw English reasoning and won on alignment, Chinese, tool use, and long context — the axes a deployable assistant actually lives or dies on. That is a *strategy*, not an accident: with a finite training budget you cannot lead on every axis, so you choose where to spend, and GLM-4 spent on the axes its users would feel rather than the ones that win headline comparisons.

### What the tool-augmented results actually prove

The All Tools numbers reward a closer look, because they make a point that's easy to miss: the gap between GLM-4 and GPT-4 on *raw* GSM8K (93.3 vs 95.6) nearly *vanishes* when both are allowed a Python interpreter (91.59 vs 92.72). That's not a coincidence. Grade-school math is exactly the kind of task where a tool removes the bottleneck — the model doesn't need to do arithmetic in its head if it can write `print(37 * 19)` — so once both models have the tool, the residual gap reflects only their *planning* (writing correct code), not their *computation*. The convergence under tools tells you that GLM-4's reasoning deficit relative to GPT-4 was substantially a *computation* deficit, which tools paper over. This is a genuinely useful diagnostic for any model: compare raw vs tool-augmented performance, and the gap that *closes* under tools was computation, while the gap that *remains* is planning and judgment. For a deployable assistant — which will have tools — the tool-augmented numbers are the ones that matter, and on those GLM-4 is essentially at parity.

The browsing win (78.08 vs 67.12) makes the complementary point. Browsing is a *planning-and-judgment* task — what to search, which result to trust, when you have enough — not a computation task, so a tool doesn't paper over a deficit; it *rewards* the model that plans better. GLM-4 winning here, after three years of browsing-specific groundwork (WebGLM), is the clearest evidence that focused, sustained investment in a capability compounds into a real edge. It's the difference between a model that *can* browse (most can, given the tool) and one that browses *well* (which takes training the judgment).

## War stories: ten alignment decisions worth stealing

### 1. Spending on authentic data instead of a bigger generator

The "authentic human prompts beat synthetic" stance is the highest-leverage data decision in the report, and it runs against the grain of a field that loves to scale synthetic data because it's cheap. The reason it's worth stealing is economic: collecting authentic human interactions is expensive and slow, but the report's claim is that the *marginal* authentic example is worth more than the *marginal* synthetic one for alignment quality. If that's true even half the time, a team pouring its budget into a synthetic-data generator is misallocating. The discipline is to A/B the two — finetune on authentic vs synthetic and measure alignment — before committing the budget, rather than defaulting to synthetic because it scales. The nuance, and the reason this isn't dogma, is that synthetic data is excellent for *capability* (teaching the model to do math, write code, follow a format) and authentic data is what matters for *alignment* (matching the messy distribution of real requests and the subtle texture of good responses). So the sophisticated position isn't "authentic good, synthetic bad" — it's "use synthetic to teach capabilities you can verify, and spend your scarce authentic-data budget on the alignment quality that synthetic can't buy." The GLM team's stance is best read as a corrective to a field that had over-rotated toward synthetic-for-everything, not as a blanket rejection of synthesis.

### 2. The three PPO stabilizers, as a checklist

Most teams that try RLHF and fail, fail because PPO collapsed and they didn't know which knob to turn. ChatGLM-RLHF's three stabilizers are a diagnostic checklist: if your policy is lurching, look at *reward variance*; if you're out of memory, look at *gradient fusion* across model-parallel shards; if your model is reward-hacking and losing general ability, look at *forgetting regularization*. Each failure mode maps to a specific fix. The general lesson is that "RLHF" is not one technique but a stack of stabilizers around a core algorithm, and the stabilizers are where the production engineering lives. This is why "we implemented RLHF" and "we have a working RLHF system" are completely different claims, separated by months of stabilizer engineering — and why so many teams report that their first RLHF attempt produced a worse model than the SFT baseline. The first attempt almost always lacks the stabilizers, so it reward-hacks or oscillates. The ChatGLM-RLHF paper is valuable precisely because it names the three stabilizers explicitly, turning "RLHF is finicky" into a concrete checklist of what to add.

### 3. Using both PPO and DPO

GLM-4 didn't pick a side in the PPO-vs-DPO debate; it used both. DPO (direct preference optimization) is simpler and more stable — it optimizes preferences directly without a separate reward model and RL loop — but PPO can squeeze more out of a good reward model. Using both lets you apply the cheaper, more stable DPO where it suffices and reserve full PPO for where it pays off. The lesson is that these aren't competing religions; they're tools with different cost-stability-ceiling tradeoffs, and a mature pipeline uses the right one per situation rather than committing to one ideologically. A practical division many teams arrive at: DPO for broad, easy-to-judge preferences where its stability shines, and PPO for cases needing optimization against a reward model that captures what a pairwise framing can't — multi-turn coherence, or a reward combining several signals. The cost of using both is maintaining two training paths; the benefit is not being boxed in by either method's weaknesses. This refusal to be ideological about methods recurs throughout the GLM lineage, and it's a healthier engineering culture than the "we are a DPO shop" or "we are a PPO shop" tribalism that's surprisingly common.

### 4. The loss-weighting bug in packed long-context training

LongAlign's loss-weighting fix addresses a bug almost everyone hits when they start packing long sequences and almost no one notices, because the model still trains and the loss still goes down — it's just learning a subtly different objective than intended (over-weighting long examples). The transferable warning: whenever you change the *batching* for throughput, sit down and re-derive what your loss is averaging over. Packing, sorting, dynamic batching — each can silently reweight your objective, and a silent objective change is the worst kind of bug because nothing crashes. The reason this class of bug is so pernicious is that every observable signal looks healthy: the loss decreases, the gradients are finite, the throughput is great, the model improves on your long-context eval. The only symptom is that the model is subtly worse on short examples than it should be — and you might not even have a short-example eval in your long-context test suite, so you never see it. The defense is a habit, not a tool: any time you touch the data-loading or batching code, write down the formula for what your loss is averaging over, before and after, and confirm they match your intent. It takes ten minutes and catches a bug that can silently cost you weeks of a degraded run.

### 5. Two thousand trajectories, not a million

AgentInstruct's 1,866 trajectories is a startlingly small dataset for the capability it unlocks. The lesson is that agent skill — the ability to plan, call a tool, read the result, and continue — is more about *quality and coverage* than *volume*. A few high-quality, reward-filtered trajectories across diverse environments teaches the pattern; a million mediocre ones mostly teach noise. When you're adding a structured skill, resist the instinct to maximize dataset size and instead maximize per-example quality and environment diversity. This runs against a deep instinct in machine learning — "more data is better" — which is true for *pretraining* (where you're estimating a broad distribution) but often false for *skill grafting* (where you're teaching a specific competence). For skills, a small, clean, diverse set teaches the pattern; a large, noisy set teaches the pattern plus a lot of noise that the small parameter budget you're willing to move can't separate. The 1,866-trajectory result is a data point worth keeping in your head as a counterweight the next time someone proposes spending three months scaling an agent dataset to a million examples — the marginal value of example 100,000 is probably near zero, and that effort is better spent on diversity and filtering.

### 6. The 1:4 ratio that protects the generalist

The mixing ratio is the actual AgentTuning technique, and it generalizes to any narrow-skill-on-generalist problem. Train on the narrow skill alone and the model forgets everything else; dilute it four-to-one with general data and the skill generalizes *and* the generalist survives. The number isn't sacred — your optimal ratio depends on the skill and the model — but the *principle* is: the dilution ratio is a hyperparameter you must tune, not an afterthought, and erring toward more general data is safer than erring toward more narrow data. The mechanism behind the cannibalization is worth understanding: finetuning is a form of catastrophic forgetting in slow motion. Every gradient step on agent data nudges the weights toward "agent" and, inevitably, slightly away from everything else. With pure agent data, those nudges accumulate until the general capabilities are overwritten. Interleaving general data resets the drift — each general example pulls the weights back toward general competence — so the net effect is "learn the agent skill while staying anchored to general ability." The 1:4 ratio is essentially a statement about how much anchoring you need per unit of new skill, and the fact that it's 4:1 (four parts anchor, one part new) tells you the anchoring need is large. This is the same dynamic as the forgetting-regularization in RLHF, just implemented through data mixing instead of a loss penalty — two different mechanisms guarding against the same failure (losing what you had while learning something new).

### 7. Autonomy as the agentic dividing line

The thing that makes All Tools an *agent* rather than a tool-augmented chatbot is that the model decides which tool to use, unprompted. This is a higher bar than function-calling, where the developer's harness decides when to invoke a tool. Training for autonomy means the model has to learn *planning* — recognizing that a request needs the browser, or that a math problem needs the interpreter — which is a different skill than executing a tool call you were told to make. The lesson for anyone building agents: the gap between "calls tools when told" and "decides to call tools" is where most of the difficulty (and the value) lives. A useful way to test whether your "agent" is actually autonomous is to give it a request that *doesn't* need a tool and see if it restrains itself. A function-calling model wrapped in a harness that always offers tools will often use one gratuitously (searching the web to answer "what's 2+2"); a genuinely autonomous model recognizes when its own knowledge suffices and answers directly. That restraint — knowing when *not* to act — is as much a part of autonomy as knowing when to act, and it's the part that's hardest to train because the supervision signal ("you should have just answered") is subtle.

### 8. Building the benchmark you need

LongBench-Chat (for long context) and AlignBench (for Chinese alignment) are benchmarks the GLM team *built* because the capabilities they were optimizing didn't have good public rulers. This is a meta-technique: if you're optimizing a capability no existing benchmark captures well, the benchmark is part of the deliverable. You cannot improve what you cannot measure, and for genuinely new capabilities the measurement often doesn't exist yet. Budget for building the eval, not just the model. There's a subtle second benefit: the benchmark you build *defines* the capability in a way that shapes your training. When the GLM team built LongBench-Chat with 10K–100K-token queries, they were implicitly deciding what "good long-context" means (following instructions buried deep in a long document, not just retrieving a needle), and that definition then guided the LongAlign data. The benchmark and the training co-evolve: a sharper benchmark produces sharper training, which is another reason the benchmark is part of the deliverable rather than an afterthought.

### 9. Merging tokenizers instead of choosing

GLM-4's vocabulary merges GLM's own byte-level BPE with OpenAI's `cl100k_base`. Rather than choosing between strong Chinese tokenization (its own) and strong English tokenization (tiktoken's), it took both. The lesson is that the tokenizer is not a place to be precious about purity — a merged vocabulary that tokenizes both languages efficiently is worth more than an elegant single-source one that handicaps one language. For a bilingual model, tokenizer efficiency in *both* languages is a capability, and merging is a cheap way to get it. The cost of a bad tokenizer is easy to underestimate: if your tokenizer splits common Chinese words into many tokens, every Chinese request costs more tokens to represent, which means less effective context, slower generation, and higher serving cost — a tax paid on every single Chinese interaction forever. A merged vocabulary that tokenizes Chinese efficiently isn't an elegance nicety; it's a permanent efficiency gain on half your traffic. The reason teams don't always do this is path dependence — you pick a tokenizer early, train on it, and changing it means retraining — which is exactly why the *time* to get the tokenizer right is before pretraining, not after. GLM's willingness to build a custom merged vocabulary rather than inherit an English-centric one off the shelf is a small decision with a large, compounding payoff.

### 10. Capabilities that compound

The deepest pattern in this whole story is that GLM-4 All Tools wasn't a 2024 breakthrough — it was the convergence of WebGLM (2023 browsing), AgentTuning (2023 trajectories), and the ChatGLM3 function-call format (2023). Each was a separate project that, individually, looked incremental. Together, three years later, they were exactly the pieces needed for autonomous tool use. The lesson for research strategy: capabilities compound, and the teams that win the agentic race are the ones that started accumulating the pieces — retrieval, tool formats, agent data — years before the payoff. You can't sprint to an agent; you assemble one. This reframes how to read a "sudden" capability announcement: when a lab ships an impressive agentic model, the right question isn't "how did they do that this year?" but "what did they ship three years ago that this is built on?" For GLM-4 All Tools the answer is WebGLM, AgentTuning, and ChatGLM3's function format — none of which looked like a frontier breakthrough at the time. The strategic implication for anyone planning a research roadmap is that the highest-leverage work often looks incremental in isolation and only reveals its value when the pieces combine. Funding and judging research purely on per-project impact systematically under-invests in exactly the compounding groundwork that produces the eventual breakthrough.

### 11. The bilingual token-mixing problem

A failure mode specific to bilingual models, and a good illustration of what RLHF fixes that SFT doesn't: GLM-4 sometimes generated responses that randomly switched between Chinese and English mid-sentence — a "token-mixing" pathology. SFT struggles with this because the training data, being authentic, occasionally contains code-switching, and the model learns it's sometimes acceptable. RLHF fixes it because human raters reliably *dislike* unwanted code-switching, so the preference signal pushes it out. The general lesson is that some failure modes are best characterized as "things humans consistently dislike but that appear in the data" — language-mixing, repetition, format drift — and these are exactly the failures RLHF is suited to, because they're easy to express as a preference and hard to express as a supervised label. When you find a pathology that's "I know it when I see it" but hard to define, it's an RLHF problem, not an SFT problem. This is a genuinely useful triage heuristic for any post-training problem. Ask: can I write down, as a supervised label, what the correct output should be? If yes — the model should output *this specific thing* — it's an SFT problem, fix it with better SFT data. If no — I can't specify the right output, but I can reliably *rank* two outputs or *recognize* a bad one — it's a preference problem, fix it with RLHF or DPO. Language-mixing, sycophancy, repetition, excessive hedging, and format drift all fall in the second bucket: you can't enumerate every acceptable response, but you know the bad ones when you see them. Matching the failure to the right tool (supervised vs preference) is half the battle; teams waste enormous effort trying to SFT their way out of preference-shaped problems, collecting ever more "correct" examples for a pathology that's about *ranking*, not *labeling*.

### 12. The open release as a capability statement

GLM-4-9B wasn't just a smaller GLM-4; it was the *open* one, shipped with the same post-training pipeline as the flagship and with the experimental 1M-context variant. Releasing the recipe (not just the weights) at 9B — a size researchers can actually finetune — is a strategic choice that the whole GLM lineage makes repeatedly: the open models are how the techniques propagate and how the ecosystem builds trust. The 1M-context variant in particular is a *capability statement* — "our long-context recipe scales this far" — even if 1M isn't a production guarantee. The lesson for a lab is that open releases at finetunable sizes are not charity; they're how your techniques become the field's defaults, and being the default is its own kind of moat. The GLM-4-9B release, like the GLM-130B release before it, reflects a consistent bet that openness at finetunable sizes compounds into ecosystem influence, and the later GLM-4.5 and GLM-4.6 open releases continue it — a strategic throughline as deliberate as any of the technical ones in this article.

## When to reach for the GLM-4 alignment playbook

There's a meta-pattern across all five techniques worth naming before the triage. Each one is a case where the *obvious* version of a step has a hidden failure mode, and the technique is the specific patch: SFT's obvious version (synthetic data) inherits the generator's blind spots; RLHF's obvious version (vanilla PPO) collapses; long-context's obvious version (extend RoPE, pack sequences) silently mis-weights the loss; agent tuning's obvious version (train on agent data) cannibalizes generality; tool use's obvious version (function-calling) isn't actually autonomous. The alignment expertise isn't knowing the five steps — everyone knows the five steps. It's knowing the five *failure modes* and their fixes. That's the difference between a team that reads the pipeline diagram and a team that has shipped the pipeline.

**Reach for these techniques when:**

- **You care about alignment quality, not just capability.** The authentic-data SFT stance and the three-stabilizer RLHF are the difference between a model that benchmarks well and one that *feels* aligned in real use. If users will interact with your model directly, this is the playbook. The signal that you need it is a model that scores well but generates responses that are subtly off — too long, too hedged, occasionally unsafe, losing the thread on turn four — the things benchmarks miss but users feel immediately.
- **You're extending context and seeing instruction-following degrade at length.** That's the LongAlign symptom. Check your packing's loss weighting first — it's the most common silent culprit.
- **You're adding a narrow skill (tools, a domain, a format) to a generalist.** Use the 1:4-style mixing ratio and tune it. Don't train on the narrow skill alone.
- **You're building an agent.** Aim for autonomous tool selection, not just function-calling, and accumulate the pieces (retrieval, tool formats, agent trajectories) deliberately.

**Skip or adapt when:**

- **You have no users yet.** The authentic-data stance assumes you can collect real human interactions. Pre-launch, you may have to bootstrap with synthetic data — just plan to replace it with authentic data as soon as you have users.
- **DPO suffices for your scale.** Full PPO with three stabilizers is a lot of machinery. If your preferences are clean and your scale is modest, DPO alone may get you most of the way with far less engineering.
- **Your context is short.** If you're not going past ~8K, the LongAlign packing machinery is solving a problem you don't have; standard finetuning is fine.
- **Your "tools" are a fixed pipeline.** If your application always calls the same tool in the same order, you don't need autonomous selection — a hard-coded harness is simpler and more reliable than a model that decides. Autonomy is a *liability* when the control flow is deterministic, because you're adding a failure mode (the model routing wrong) to solve a problem you don't have. Reach for autonomous tool use only when the request space is genuinely open-ended enough that you *can't* enumerate the right tool sequence in advance.
- **You're working in one language.** A chunk of GLM-4's engineering — the merged bilingual tokenizer, the token-mixing RLHF fix, the 50/50 data balance — is specifically about being good in two languages at once. A monolingual model skips all of it, and copying those choices would be solving a problem you don't have.

A closing caution that applies to every technique here: these are the choices of a team building a *flagship general assistant* with a large budget and a real user base. If you're finetuning a small model for a narrow task, most of this machinery is overkill, and the right move is to take the *principles* (authentic data beats synthetic for alignment; mix narrow skills with general data; re-derive your loss after changing batching) without the full apparatus. The principles are scale-free; the apparatus is not.

The unifying thread is that GLM-4's alignment story is a series of *non-obvious failure modes with specific fixes* — synthetic data's hidden blind spots, PPO's collapse modes, packing's silent reweighting, agent data's cannibalization of generality. The next article in the series moves to GLM-4.5, where these alignment techniques get industrialized into the expert-distillation and agentic-RL pipeline that produces a frontier model. But the habit — name the failure mode, fix it specifically — is the same one [GLM-130B](/blog/machine-learning/mlops/engineering-glm-130b) showed on the engineering side.

If you take one thing from this article, make it the framing rather than any single technique: post-training is not a recipe you follow but a *gauntlet of failure modes you anticipate*. The five steps are common knowledge; the value is in knowing that synthetic data hides blind spots, that PPO collapses without stabilizers, that packing reweights your loss, that agent data cannibalizes generality, and that function-calling isn't autonomy — and in having a specific fix ready for each. That's what separates a team that can recite the alignment pipeline from a team that has shipped an aligned model. The GLM-4 report is, read correctly, a field guide to the quiet failures, and the next article will show those same instincts industrialized at frontier scale, where the failure modes get harder and the fixes get more elaborate but the underlying discipline — name it, fix it specifically, write it down — never changes.
