Every team that turns on speculative decoding discovers the same thing in the same order. First, the demo is thrilling: time per output token drops by half, the stream feels instant, and somebody screenshots the latency graph for the standup. Then they try it on their own model — a fine-tuned Qwen3, an internal 30B MoE, a code model with a custom tokenizer — and the speedup collapses to 1.1x. Nothing is broken. The serving stack is doing exactly what it promises. The problem is that the draft model is bad, and there is no button in the serving stack that fixes a bad drafter.

This is the asymmetry nobody warns you about. The *serving* side of speculative decoding is a solved engineering problem with excellent open-source implementations: [SGLang](https://docs.sglang.ai/), vLLM and TensorRT-LLM all ship tree-based verification, rejection sampling and CUDA-graph-captured draft loops. The *training* side — producing a drafter whose distribution is close enough to your target model that it survives verification — was, until recently, a pile of half-maintained research repos. The EAGLE authors published reference code. Everyone else forked it, patched it for their architecture, and quietly gave up when their MoE target didn't fit the assumptions.

SpecForge is the SGLang team's answer to that gap. Its own README is blunt about why it exists: "We have seen many open-source projects for speculative decoding, but most of them are not well-maintained or not directly compatible with SGLang." The pitch is narrow and, I think, correct — a training framework whose output loads into the serving engine with *no porting step at all*, because the same engine that will serve the model is also the one that captured the training features.

![The SpecForge loop: dataset, target-model regeneration, feature capture in a patched SGLang server, draft training, export, and SGLang serving](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-1.webp)

The diagram above is the mental model for this entire article: SGLang appears **three times** in one loop. It runs as a local engine when you pre-compute hidden states offline. It runs as a patched server that streams features into the trainer when you train online. And it runs as the serving runtime that loads the exported draft. Everything else in SpecForge — the algorithms, the loss functions, the distributed plumbing — exists to move data around that loop without ever letting the target model into the trainer's own process.

This post is a code-level tour. I cloned the repository at `v0.2.0` and read it, so the file paths, line-level mechanics and default values below are the real ones rather than a paraphrase of the docs. We will walk the EAGLE3 draft architecture, the training-time test loop and the peculiar attention mask it needs, the loss that optimizes acceptance directly, the five other drafting algorithms, the three training topologies, and finally the export boundary where a beautifully-trained checkpoint can still fail silently.

## Why the drafter is the hard half

The instinct most engineers bring to speculative decoding is that the drafter is "just a small model," and that a small model is easy. That instinct is wrong in a specific and expensive way.

| Assumption | The naive view | What actually happens |
| --- | --- | --- |
| "Any small model works as a drafter" | Grab a 0.5B from the same family | Different tokenizer or different fine-tune means acceptance collapses; the drafter must track *this* target's distribution, not the family's |
| "Bigger drafter, better acceptance" | Scale the draft model up | Draft cost is paid on *every* step. EAGLE3's drafter is a **single decoder layer**; a bigger one wins acceptance and loses wall-clock |
| "Train on any chat corpus" | ShareGPT and call it done | Production checkpoints are trained on answers **regenerated by the target model** — the drafter is imitating the target, not humans |
| "Acceptance rate is the metric" | Optimize token accuracy | What buys speedup is *accepted length* per verify, and it varies 2x across domains: about 2.8 on chat, about 5.4 on math |
| "Train it, then port it to the server" | Convert weights afterwards | Serving weight loading is **fail-silent** — an unrecognized key is skipped and the server still starts, quietly serving garbage |
| "The trainer needs the target model" | Load a 671B target next to the drafter | The trainer only ever holds the drafter and a frozen `lm_head`; features come from outside |

The last two rows are the ones that generate production incidents, and we will come back to both.

There is one more structural fact worth stating early, because it reframes the whole exercise. Speculative decoding is **lossless**: the output distribution is provably identical to the target model's, because rejected tokens are resampled from a corrected distribution. That means a bad drafter cannot corrupt your outputs — it can only fail to speed them up. This is unusually forgiving. You can ship a mediocre drafter with zero quality risk, then improve it offline. Training a drafter is therefore a pure throughput investment, and it should be evaluated like one.

## 1. Speculative decoding in one round

Autoregressive decoding is memory-bound. To emit one token, the GPU must stream every weight of the model out of HBM, do a trivial amount of arithmetic per weight, and throw the weights away. At batch size 1 on an 8B model, an H100 sits at single-digit utilization while the memory bus saturates. The FLOPs are free; the bandwidth is not.

Speculative decoding spends those free FLOPs. A cheap drafter proposes several tokens; the target model then scores *all* of them in a single forward pass, because scoring $k$ positions costs almost exactly what scoring one position costs when you are bandwidth-bound. The tokens that match what the target would have produced are kept.

![One speculative round: a target forward, seven cheap draft steps, tree expansion, one verify pass, prefix acceptance and resampling of the first mismatch](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-2.webp)

The acceptance rule is what makes this lossless. Let $p$ be the target distribution at some position and $q$ the draft distribution. Sample $x \sim q$. Accept it with probability $\min(1, p(x)/q(x))$. If it is rejected, sample the replacement from the normalized residual $\max(0, p - q)$. Leviathan et al. ([2211.17192](https://arxiv.org/abs/2211.17192)) prove that the resulting samples are distributed exactly as $p$. The expected number of accepted tokens at a position is the overlap between the two distributions:

$$
\alpha = \sum_{v} \min\big(p(v),\, q(v)\big)
$$

This single quantity is the north star of the entire framework. Remember it — SpecForge computes it during training, logs it, and one of its loss functions optimizes it directly.

Now the economics. If a round proposes $k$ tokens and the expected accepted length is $\tau$, then per round you pay one target forward plus $k$ draft forwards, and you receive $\tau$ tokens. The speedup over plain decoding is roughly

$$
S = \frac{\tau}{1 + k \cdot c}
$$

where $c$ is the draft cost as a fraction of a target forward. This is why the drafter's *size* matters so much: EAGLE-style drafters are one decoder layer, so $c$ is a few percent and the denominator stays near 1. It is also why acceptance length dominates everything. Going from $\tau = 2.8$ to $\tau = 5.4$ nearly doubles your speedup, and that gap is exactly the gap between chat traffic and math traffic on the same checkpoint.

Two practical consequences follow immediately. First, speculative decoding is a **latency** optimization, and its value degrades as batch size grows — at high load the GPU is no longer bandwidth-starved, so the free FLOPs you were spending are no longer free. (I wrote about that tradeoff in detail in [Speculative Decoding in Production](/blog/machine-learning/model-serving/speculative-decoding-in-production).) Second, since $\tau$ is domain-dependent, a drafter trained on chat data and deployed on a coding assistant is leaving most of the win on the floor.

## 2. Two lineages of drafters

SpecForge does not implement "speculative decoding." It implements six trainable drafting algorithms from two distinct research lineages, and understanding which lineage you are in explains most of the configuration surface.

![The two lineages: feature-level autoregression from EAGLE through EAGLE-3, and block-diffusion drafting from DFlash, each with three refinements](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-3.webp)

**The feature-level autoregression lineage** starts with EAGLE ([2401.15077](https://arxiv.org/abs/2401.15077)). Its insight is that predicting the target's *hidden features* is an easier learning problem than predicting its tokens, because features are continuous and carry more information than a sampled token id. EAGLE's drafter therefore autoregresses at the feature level, and resolves the inherent ambiguity of "which token was actually sampled" by feeding the token sequence shifted one step forward. On LLaMA2-Chat 70B it reported 2.7x-3.5x latency speedups while provably preserving the output distribution.

EAGLE-2 ([2406.16858](https://arxiv.org/abs/2406.16858)) kept the architecture and changed the draft *tree*. The original used a static tree shape, implicitly assuming acceptance depends only on position. EAGLE-2 observed that acceptance is strongly context-dependent and that a well-calibrated drafter's own confidence approximates its acceptance rate closely. So it expands and reranks the tree dynamically using that confidence, reaching 3.05x-4.26x — 20-40% faster than EAGLE-1 with no retraining of the architecture.

EAGLE-3 ([2503.01840](https://arxiv.org/abs/2503.01840)) is the one SpecForge is built around, and it made two changes that matter enormously for a *training* framework. It dropped the feature-regression loss and predicts tokens directly, and it fuses hidden states from three depths of the target instead of only the top layer. The paper's headline finding is not the raw speedup (up to 6.5x) but a scaling claim: earlier EAGLEs *stopped improving* when you gave them more training data, because the feature-prediction constraint bottlenecked them. EAGLE-3 removes that ceiling. A drafter that gets better with more data is precisely the thing that justifies building industrial training infrastructure — which is the entire reason SpecForge exists.

**The block-diffusion lineage** is newer and structurally different. DFlash ([2602.06036](https://arxiv.org/abs/2602.06036), ICML 2026) replaces autoregressive drafting with a lightweight block diffusion model that emits an entire block of draft tokens in *one* forward pass, conditioned on context features extracted from the target. Its claim is over 6x lossless acceleration and up to 2.5x higher speedup than EAGLE-3 — the gain coming from the fact that drafting $k$ tokens no longer costs $k$ sequential forwards. Domino ([2605.29707](https://arxiv.org/abs/2605.29707)) adds a recurrent correction on top; DSpark ([2607.05147](https://arxiv.org/abs/2607.05147)) makes the block semi-autoregressive with a learned confidence schedule.

The two remaining entries are refinements rather than lineages. P-EAGLE ([2602.01469](https://arxiv.org/abs/2602.01469)) restructures EAGLE3's *training* to run in one packed forward. LK loss ([2602.23881](https://arxiv.org/abs/2602.23881)) and D-PACE ([2605.18810](https://arxiv.org/abs/2605.18810)) are objectives, not architectures.

Keep the split in mind: **one token per draft step** on the left, **sixteen tokens per forward** on the right. Nearly every difference in configuration, masking, and loss follows from that one distinction.

## 3. Inside the EAGLE3 draft model

Let us look at what is actually instantiated. The draft model registered as `LlamaForCausalLMEagle3` is startlingly small — and the algorithm registration hard-pins it to exactly one layer via `fixed_override_values=(("num_hidden_layers", 1),)`.

```python
# specforge/modeling/draft/llama3_eagle.py
@register_draft
class LlamaForCausalLMEagle3(Eagle3DraftModel):
    config_class = LlamaConfig

    def __init__(self, config, quant_config=None, attention_backend="sdpa") -> None:
        self.vocab_size = config.vocab_size
        self.draft_vocab_size = config.draft_vocab_size
        self.embed_tokens = nn.Embedding(
            config.vocab_size, config.hidden_size, config.pad_token_id
        )
        self.midlayer = LlamaDecoderLayer(config, attention_backend=attention_backend)

        self.target_hidden_size = getattr(config, "target_hidden_size", config.hidden_size)
        self.fc = torch.nn.Linear(self.target_hidden_size * 3, config.hidden_size, bias=False)

        self.norm = LlamaRMSNorm(config.hidden_size, eps=config.rms_norm_eps)
        self.lm_head = nn.Linear(config.hidden_size, config.draft_vocab_size, bias=False)

        # vocab-compression buffers, exported with the checkpoint
        self.register_buffer("t2d", torch.ones(self.vocab_size, dtype=torch.bool))
        self.register_buffer("d2t", torch.zeros(self.draft_vocab_size, dtype=torch.int64))
```

That is the whole model: an embedding table, **one** decoder layer, a fusion projection, a norm, and a head over a compressed vocabulary.

![The EAGLE3 drafter: three target layers fused by fc, concatenated with the token embedding, consumed by one decoder layer with 2h-wide q/k/v projections](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-4.webp)

### Which target layers get tapped

The `fc` projection is `3 * target_hidden_size -> hidden_size`, because EAGLE3 fuses hidden states captured at three depths of the target. The defaults are computed from the target's layer count:

```python
# specforge/algorithms/model_providers.py
if layers is None:
    target_config = getattr(target_config, "text_config", target_config)
    num_layers = int(target_config.num_hidden_layers)
    layers = [1, num_layers // 2 - 1, num_layers - 4]
```

For a 36-layer Qwen3-8B that is layers 1, 17 and 32 — deliberately a *low*, a *mid* and a *high* representation. The intuition is that a single top-layer feature has already committed to a token; shallower features still carry syntactic and positional information that helps the drafter guess several steps ahead. Exactly three layers are enforced, and the capture contract validates that the stored tensor width equals `len(aux_layers) * target_hidden_size`, so a mismatched capture is rejected rather than silently reshaped.

### The double-width mouth

Here is the part that surprises people reading the code for the first time. The draft decoder layer is *not* a standard Llama layer. It consumes two streams — the token embedding and the fused hidden state — and its attention projections are twice as wide as you would expect:

```python
# specforge/modeling/draft/llama3_eagle.py — LlamaAttention.__init__
self.q_proj = nn.Linear(self.hidden_size * 2, self.num_heads * self.head_dim, bias=False)
self.k_proj = nn.Linear(self.hidden_size * 2, self.num_key_value_heads * self.head_dim, bias=False)
self.v_proj = nn.Linear(self.hidden_size * 2, self.num_key_value_heads * self.head_dim, bias=False)
self.o_proj = nn.Linear(self.num_heads * self.head_dim, self.hidden_size, bias=False)
```

And the forward normalizes each stream *separately* before concatenating them:

```python
# specforge/modeling/draft/llama3_eagle.py — LlamaDecoderLayer.forward
residual = hidden_states
hidden_states = self.hidden_norm(hidden_states)
input_emb = self.input_layernorm(input_emb)

hidden_states = torch.cat((input_emb, hidden_states), dim=-1)   # 2h wide
hidden_states = self.self_attn(
    cache_hidden=cache_hidden, hidden_states=hidden_states, ...
)
hidden_states = residual + hidden_states     # residual carries only the h-dim stream
```

Two details are load-bearing. There are **two input norms**, not one, because the embedding stream and the target-feature stream have wildly different scales and sharing a norm would let one drown the other. And the residual connection carries *only* the hidden stream — the embedding is injected fresh at every step rather than accumulated. That is what makes the layer a function of "where we are in feature space" plus "which token was actually emitted," which is precisely EAGLE's original insight about resolving feature-level ambiguity.

### The embedding is borrowed, not learned

The drafter's embedding table covers the *full* target vocabulary and is copied straight out of the target model's safetensors, then frozen:

```python
draft_model.load_embedding(target_model_path, embedding_key="model.embed_tokens.weight")
draft_model.freeze_embedding()
```

This has a consequence that shows up much later at export time: since the embedding is frozen and identical to the target's, it is deliberately **omitted** from saved checkpoints, and the serving runtime reuses the target's own table. The checkpoint machinery whitelists those missing keys explicitly, and it fingerprints the omitted frozen tensors so that resuming against a *different* target model is rejected rather than silently producing a drafter wired to the wrong embeddings.

## 4. Training-time test: teaching the drafter to survive its own mistakes

Here is the central problem of training any autoregressive drafter. At serving time the drafter runs several steps in a row, and from step 2 onward it is consuming **its own** hidden states — outputs it produced, not ground truth from the target. If you train it only on ground-truth features, you get the classic exposure-bias failure: step 1 is excellent, step 2 is mediocre, step 3 is noise, and your accepted length never exceeds about 2.

EAGLE-3's answer is *training-time test* (TTT): simulate the multi-step rollout during training, so the drafter learns to recover from its own drift. SpecForge implements this as a recursive loop with `ttt_length = 7` by default.

### The loop

```python
# specforge/algorithms/eagle3/model.py — OnlineEagle3Model.forward
for idx in range(self.length):                       # self.length = ttt_length = 7
    state = adapter.step_view(idx=idx, ttt_length=self.length, ...)

    # 5.1 embed the (shifted) input ids
    inputs_embeds = self.draft_model.embed_input_ids(state.input_ids)
    inputs_embeds = inputs_embeds.to(hidden_states.dtype)

    # 5.2 run the one-layer backbone
    hidden_states_out = self.draft_model.backbone(
        input_embeds=inputs_embeds,
        hidden_states=state.hidden_states,
        cache_hidden=cache_hidden,
        attention_mask=state.attention_mask,
        position_ids=state.position_ids,
        past_key_values=past_key_values,
        use_cache=True,
    )
    hidden_states = hidden_states_out                # <-- recycle its own output

    logits = self.draft_model.compute_logits(hidden_states)
    (acc, acceptance_rate, loss, ...) = self._acc_and_loss(...)
    plosses.append(loss)

    if not is_last:
        # shift the token stream and the masks left by one
        global_input_ids = padding(global_input_ids, left=False)
        position_mask   = padding(position_mask, left=False)
        loss_mask       = padding(loss_mask, left=False)
```

The single most important line is `hidden_states = hidden_states_out`. The drafter's own output becomes the next step's input feature — exactly the serving-time condition. Meanwhile the *token* stream shifts left by one each step (`padding(..., left=False)` is `cat(tensor[:, 1:], zeros)`), so at step $k$ position $i$ is fed token $x_{i+k}$ and asked to predict $x_{i+k+1}$. The teacher tensors are pre-padded by `ttt_length` positions and sliced with an offset per step, which is how a single precomputed teacher distribution serves all seven steps.

The fusion projection is applied **once**, outside the loop — `hidden_states = self.draft_model.project_hidden_states(hidden_states)` — because the three captured target layers are a property of the real context, not of the rollout step.

### The mask this needs

Now the subtle part. At step $k$, what should a query attend to? It needs the real prefix (the actual conversation), and it needs the recycled draft states from steps ${1..k-1}$ — but *only at its own position*, because step $j$'s state at position $i$ is a prediction about position $i$, not a new token in the sequence.

![The TTT attention mask: causal against the step-0 KV block, diagonal-only against each recycled step](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-5.webp)

So the mask is "causal to step-0 keys, diagonal-only to every later step's keys." That is not expressible as a standard causal mask, and SpecForge implements it three different ways depending on the attention backend.

The SDPA path does it by hand, with a materialized causal mask for the step-0 block and an explicit element-wise product for each recycled step:

```python
# specforge/modeling/draft/llama3_eagle.py — LlamaAttention.forward
cache_hidden[0] = cache_hidden[0] + [key_states]
cache_hidden[1] = cache_hidden[1] + [value_states]
k0, v0 = cache_hidden[0][0], cache_hidden[1][0]

attn_weights = torch.matmul(query_states, k0.transpose(2, 3)) / math.sqrt(self.head_dim)
attn_weights = attn_weights + attention_mask        # causal + padding, step-0 block only

for i in range(1, lck):
    ki = cache_hidden[0][i]
    attn_weightsi = (query_states * ki).sum(-1) / math.sqrt(self.head_dim)   # diagonal!
    attn_weights = torch.cat((attn_weights, attn_weightsi[..., None]), dim=-1)

attn_weights = nn.functional.softmax(attn_weights, dim=-1, dtype=torch.float32).to(query_states.dtype)
```

The softmax therefore runs over `q_len + (lck - 1)` logits per query: a full causal row against the real prefix, plus exactly **one scalar per previous TTT step**. It is a neat trick — the recycled states participate in the softmax normalization without ever materializing a full attention block.

The FlexAttention path expresses the same semantics declaratively, which is much cleaner and compiles to a fused block-sparse kernel:

```python
# specforge/modeling/draft/flex_attention.py
def generate_eagle3_mask(seq_lengths, Q_LEN: int, KV_LEN: int, lck: int = 0):
    def causal_mask(b, h, q_idx, kv_idx):
        causal = q_idx >= kv_idx
        padding = (kv_idx < seq_lengths[b]) & (q_idx < seq_lengths[b])
        return causal & padding

    def suffix_mask(b, h, q_idx, kv_idx):
        suffix   = kv_idx >= Q_LEN                      # keys from recycled steps
        padding  = kv_idx % Q_LEN < seq_lengths[b]
        diagonal = (kv_idx - q_idx) % Q_LEN == 0        # same position only
        return suffix & padding & diagonal

    mask_mod = or_masks(causal_mask, suffix_mask)
    mask_mod.__name__ = f"eagle3_mask_Q_{Q_LEN}_KV_{KV_LEN}_lck_{lck}"
    return mask_mod
```

The KV cache is laid out as `[step0 | step1 | ... | step_{k-1}]`, each block `Q_LEN` wide, which is why the diagonal test is modular arithmetic. The `lck` value is baked into the function's `__name__` because the mask shape changes every TTT step and torch.dynamo keys recompilation on it — the file raises `dynamo.config.recompile_limit = 64` for exactly this reason.

One more shrinking detail that is easy to miss: each left-shift pushes a garbage zero token into the tail, so the valid length must shrink by one per step. The FlexAttention path does `seq_lengths -= lck` before building the block mask; the SDPA path achieves the same thing through the shifted loss masks. Getting this wrong trains the drafter on padding.

There is also a flash-attention path that computes exact attention over the step-0 block with `flash_attn_varlen` and merges each diagonal suffix analytically via streaming log-sum-exp, with a hand-written backward — and a USP variant that layers Ulysses all-to-all and ring attention on top for sequence-parallel training.

### Why the loss decays per step

Each of the seven steps produces its own loss, and they are combined with an exponential decay:

```python
# specforge/training/strategies/base.py
weights = [self.ploss_decay ** i for i in range(len(plosses))]     # ploss_decay = 0.8
loss = sum(weights[i] * plosses[i] for i in range(len(plosses)))
```

Step 0 gets weight 1.0, step 6 gets $0.8^6 \approx 0.26$. This is a deliberate statement about what matters. Deep steps are harder (more accumulated drift) and rarer (you only reach draft depth 6 if the first five tokens were all accepted), so letting them dominate the gradient would trade away the near-certain wins for speculative ones. The 0.8 factor roughly tracks how often each depth is actually reached in practice.

## 5. The loss: from KL proxy to acceptance itself

The per-step loss is a soft-label cross-entropy against the target's distribution, restricted to the draft vocabulary. The reference implementation is compact enough to read directly:

```python
# specforge/core/loss.py
@torch.compile(dynamic=None)
def _compute_loss(logits, target_p, position_mask):
    logits = logits.float()
    out_logp = nn.LogSoftmax(dim=2)(logits)
    plogp = target_p * out_logp
    loss = -torch.sum(position_mask * plogp, 2).mean()
    return loss
```

Since the teacher's entropy is constant with respect to the draft parameters, this is gradient-equivalent to $\mathrm{KL}(p_{\text{target}} \Vert p_{\text{draft}})$, and the code calls it `kl_loss`. In production it runs through a custom Triton autograd function (`LogSoftmaxLoss`) that streams online max and sum-exp over vocabulary chunks, skips masked rows early, and — borrowing the trick the file header credits to Liger-Kernel — writes the backward gradient **in place over the saved logits buffer** rather than allocating a second tensor.

### The proxy problem

Here is the conceptual issue. KL is a *proxy*. What actually determines speedup is the acceptance rate $\alpha = \sum_v \min(p(v), q(v))$, and minimizing KL does not maximize $\alpha$ — a drafter can reduce KL by better modeling the tail of the distribution, which contributes nothing to acceptance because those tokens are rarely sampled.

LK loss attacks this directly. The acceptance quantity is computed exactly:

```python
# specforge/core/lk_loss.py
def expected_acceptance_rate(draft_p, target_p_on_draft, ...):
    return torch.minimum(target_p_on_draft, draft_p).sum(-1)
```

Note it uses `target_p_on_draft` — the target's *unnormalized* probabilities restricted to the draft vocabulary — rather than the renormalized `target_p`. That is the honest choice: probability mass the target puts on tokens outside the 32k draft vocabulary is mass the drafter can never capture, and it correctly caps acceptance below 1.

```python
def compute_lk_loss(kl_loss, acceptance_rate, log_acceptance_rate, lk_loss_type, kl_scale, kl_decay):
    if lk_loss_type == "alpha":
        return -log_acceptance_rate
    if lk_loss_type == "lambda":
        acc_det = acceptance_rate.detach()
        kl_weight = kl_scale * torch.exp(-kl_decay * acc_det)
        return kl_weight * kl_loss + (1 - kl_weight) * (1 - acceptance_rate)
```

![LK-lambda: the blend weight decays exponentially with acceptance, handing the objective from KL to acceptance as training progresses](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-6.webp)

The `lambda` mode is the interesting one, and it is a nice piece of curriculum design. Early in training acceptance is near zero, so $\exp(-\text{decay} \cdot \alpha) \approx 1$ and the objective is essentially pure KL — dense, smooth gradients over all 32,000 logits, which is what you need when the model knows nothing. As acceptance climbs, the weight decays and the objective slides toward directly maximizing $\sum_v \min(p,q)$. With the defaults (`kl_scale = 1.0`, `kl_decay = 1.0`), acceptance 0.10 gives a KL weight of 0.90; acceptance 0.70 gives 0.50.

LK loss is **off by default** (`lk_loss_type = None`). When it is off, acceptance is still computed on every step under `no_grad` purely as a logged metric — which is the right default, because it means every run tells you the number you actually care about even when you are not optimizing it.

### What gets logged

Per TTT step the trainer emits accuracy (top-1, translated back into target-vocabulary space through `d2t`), acceptance rate, and the weighted loss. Note the subtle denominator difference: accuracy is normalized by `loss_mask` (all supervised positions) while the loss is normalized by `position_mask` (supervised **and** representable in the draft vocab). Watching accuracy per TTT depth is the single most useful training diagnostic — a healthy run shows depth-0 accuracy climbing fast and the deeper depths trailing behind at a roughly constant ratio. If depth 3+ accuracy stays flat while depth 0 improves, your TTT loop is not doing its job.

## 6. A 32,000-token vocabulary carved out of 151,936

The drafter's output head does not cover the target's full vocabulary. For a Qwen3 target with 151,936 tokens, the draft head has 32,000 rows — a 4.7x reduction in the largest matrix in the model.

![Draft vocabulary compression: frequency ranking selects 32,000 tokens, t2d is the membership mask, d2t stores offsets rather than ids](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-7.webp)

The selection is pure frequency counting over the training corpus, restricted to tokens that actually carry loss:

```python
# specforge/data/preprocessing.py
total_frequency = sum(token_dict.values())
top_N = token_dict.most_common(draft_vocab_size)
top_N_frequency_sum = sum(freq for key, freq in top_N)
print(f"top {draft_vocab_size} token frequency ratio: {top_N_ratio:.2%}")

used_tokens = [key for key, freq in top_N]
used_tokens.sort()

d2t = [used_tokens[i] - i for i in range(len(used_tokens))]
t2d = [i in used_tokens for i in range(target_vocab_size)]
```

Two representation choices are worth pausing on. `t2d` is a boolean mask over the *target* vocabulary, used to slice teacher logits into draft space (`target[..., t2d]`) and to decide which positions can be supervised at all. `d2t` stores **offsets**, not absolute ids: `target_id = draft_id + d2t[draft_id]`. Storing the difference means the identity mapping is all zeros, which is why the constructor can initialize `t2d = ones, d2t = zeros` as a valid default for the case where draft vocab equals target vocab.

The loss masking consequence is important and easy to overlook:

```python
target_token_ids = target_head.argmax(-1)
target_mask   = t2d[target_token_ids][..., None].int()
position_mask = target_mask * loss_mask
```

A position whose true next token is **not** in the draft vocabulary is dropped from the loss entirely. This is correct — you cannot teach the drafter to emit a token it has no output row for — but it means the effective supervision density depends on how well your 32k covers your domain. A drafter trained on general chat and deployed on code will find a larger fraction of positions unrepresentable, which shows up as depressed acceptance before you ever look at the model quality. If you are training for a specialized domain, regenerate the vocabulary mapping from *that* domain's data.

The mapping is derived automatically and cached under a hash of the dataset files, sizes, mtimes and vocab sizes, with only rank 0 persisting the `.pt` via an atomic replace. In disaggregated runs it must be supplied explicitly as a shared path, precisely so a producer and a consumer cannot independently derive two different mappings.

## 7. Six drafters, one trainer

The README's method table lists six entries. They share the trainer spine — the same data pipeline, the same FSDP wrapping, the same checkpoint machinery — and diverge on two axes that matter: **how many tokens one draft forward produces**, and **what supervises it**.

![Six drafters compared across draft layers, tokens per draft forward, training pass and supervision signal](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-8.webp)

### DFlash: drafting a whole block at once

DFlash abandons autoregression inside the draft. Its model is a stack of full decoder layers (1-5, typically 5 for an 8B target) with a hybrid attention: queries come only from a block of "noise" slots, while keys and values come from the union of projected target context and the block itself.

```python
# specforge/modeling/draft/dflash.py — Qwen3DFlashAttention.forward
q = self.q_proj(hidden_states)          # queries: the draft block only
k_ctx   = self.k_proj(target_hidden)    # keys over target context features
k_noise = self.k_proj(hidden_states)    # keys over the block itself
v_ctx   = self.v_proj(target_hidden)
v_noise = self.v_proj(hidden_states)
k = torch.cat([k_ctx, k_noise], dim=1)
```

Crucially, `self.is_causal = False` — attention *within* the block is bidirectional, exactly as in a masked-denoising or diffusion model. That is what lets one forward pass fill all sixteen slots simultaneously instead of sequentially.

At serving time each round builds a block whose slot 0 holds the last accepted token and whose remaining slots hold `mask_token_id`. One draft forward fills them; the target verifies; the matching prefix is accepted and both KV caches are cropped:

```python
# specforge/modeling/draft/dflash.py — spec_generate
while start < max_length:
    block_output_ids = output_ids[:, start : start + block_size].clone()
    noise_embedding = target.model.embed_tokens(block_output_ids)
    draft_hidden = self(
        target_hidden=target_hidden,
        noise_embedding=noise_embedding,
        position_ids=...,
        past_key_values=past_key_values_draft,
        use_cache=True,
        is_causal=False,
    )
    past_key_values_draft.crop(start)
    block_output_ids[:, 1:] = self._sample_draft_tokens(target, draft_hidden, block_output_ids)
```

Note `_sample_draft_tokens` calls the target's `lm_head` on the whole block suffix at once — "DFlash predicts the whole suffix in one LM-head call." The drafter borrows the target's embedding table *and* its output head rather than learning its own, so there is no draft-vocabulary compression in this family at all.

**Training** is where the design gets clever. Instead of one block per sample, SpecForge samples up to `num_anchors = 512` random loss-masked positions and builds an independent block at each one, then runs them all in a single forward disambiguated purely by the attention mask:

![DFlash training packs 512 anchor blocks into one forward, with three mask rules keeping them mutually invisible](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-10.webp)

```python
# specforge/algorithms/common/dflash_family_model.py
def create_dflash_block_mask(...):
    """KV: [Context (S tokens) | Block_0 | ... | Block_{n-1}]
       Q:  [Block_0 | ... | Block_{n-1}]
       1. Each block sees context strictly before its anchor (kv_idx < anchor_pos).
       2. Intra-block attention is bidirectional.
       3. Different blocks are invisible to each other.
       4. Invalid blocks (block_keep_mask=False) see nothing."""
    def dflash_mask_mod(b, h, q_idx, kv_idx):
        q_block_id = q_idx // block_size
        mask_context = is_context & (kv_idx < anchor_pos)   # strictly before the anchor
        is_draft = kv_idx >= S
        kv_block_id = (kv_idx - S) // block_size
        mask_draft = is_draft & (q_block_id == kv_block_id)
```

The "strictly before" is not pedantry — the code comment explains it "matches inference where `target_hidden[anchor_pos]` is not available as context." A block that peeked at its own anchor's target feature would be training under conditions that never occur at serving time, and would over-report acceptance.

The arithmetic is striking: 512 anchors × 16 slots = **8,192 supervised draft positions per sample**, and that number is independent of sequence length. The published Domino benchmark measures the cost precisely — roughly 53 ms fixed plus 0.73 ms per anchor, so 256 anchors yields about 2,130 anchor-updates per second versus about 1,280 at 64 anchors.

Two loss options exist. The default is masked token-level cross-entropy with optional exponential position decay $\exp(-(k-1)/\gamma)$, where recipes use $\gamma = 7$. The alternative is **D-PACE**, which reweights each position's CE by a *detached* estimate of how much accepted continuation it buys:

```python
smooth = (1.0 - self.dpace_alpha) * prob + self.dpace_alpha
prefix = torch.cumprod(smooth, dim=-1)                    # cumulative confidence
suffix = torch.flip(torch.cumsum(torch.flip(prefix * binary_mask, dims=[-1]), dim=-1), dims=[-1])
```

The full D-PACE weight is the suffix sum of smoothed cumulative acceptance probabilities — the expected accepted continuation from that position. It is the same philosophical move as LK loss: stop optimizing per-token likelihood, start optimizing the thing that converts to speedup. Note that none of the shipped example configs actually enable it; it is available via `training.loss_type: dpace`.

### Domino: a GRU that restores sequential conditioning

Block-parallel drafting has an obvious weakness. Slot 5 is predicted without knowing what slot 4 turned out to be, so the block is internally inconsistent in exactly the way autoregression prevents. Domino fixes this without giving up the parallel backbone pass.

![Domino: one parallel backbone pass, then a GRU-driven residual corrects the block logits position by position](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-9.webp)

```python
# specforge/modeling/draft/domino.py
self.prefix_gru = nn.GRU(
    input_size=config.hidden_size, hidden_size=self.gru_hidden_dim,   # 1024
    num_layers=1, batch_first=True, bias=False,
)
in_dim = config.hidden_size + self.gru_hidden_dim
self.embed_proj = nn.Sequential(
    nn.Linear(in_dim, self.emb_dim, bias=False),      # emb_dim = 256
    nn.SiLU(),
    nn.Linear(self.emb_dim, config.vocab_size, bias=False),
)
```

The GRU runs over the embeddings of previously-emitted tokens within the block; its state is concatenated with the block hidden state and projected to vocabulary size, then **added as a residual** on top of the base `lm_head` logits — but only from `suffix_start` onward:

```python
z_n = hidden_states[:, :, self.suffix_start :, :]
concat_features = torch.cat([z_n, prefix_states], dim=-1)
logits_e = self.embed_proj(concat_features)
prefix_logits = base_logits[:, :, : self.suffix_start, :]
suffix_logits = base_logits[:, :, self.suffix_start :, :] + logits_e
return torch.cat([prefix_logits, suffix_logits], dim=2)
```

The first `pure_draft_prefix_len` positions stay pure DFlash because at inference the GRU has no reliable input for them yet. At serving time the block backbone still runs once, but token sampling within the block becomes sequential — each position embeds what has been emitted so far and re-applies the corrected head. That is one expensive forward plus $B-1$ cheap head passes, which is a very different cost profile from $B$ backbone forwards.

The training schedule is the part I find most elegant:

```python
final_loss = final_num / valid_token_count
base_loss  = base_num / valid_token_count
loss = (1.0 - lambda_base) * final_loss + lambda_base * base_loss
```

`lambda_base` decays linearly from 1.0 to 0 over the first `total_steps * lambda_base_decay_ratio` optimizer updates. So training begins by optimizing *only* the plain DFlash backbone (the GRU contributes nothing to the gradient), then progressively hands the objective over to the corrected head. You train the foundation first, then the refinement — a curriculum that avoids the GRU learning to compensate for a backbone that is itself still garbage.

### DSpark: a cheap Markov correction plus a confidence head

DSpark takes the same DFlash backbone and adds two heads. The first is a low-rank previous-token logit bias that makes block generation semi-autoregressive at almost no cost:

```python
# specforge/modeling/draft/dspark.py
class VanillaMarkovHead(nn.Module):
    """Low-rank previous-token logit bias used by DSpark."""
    def __init__(self, *, vocab_size: int, markov_rank: int):     # markov_rank = 256
        self.markov_w1 = nn.Embedding(self.vocab_size, self.markov_rank)
        self.markov_w2 = nn.Linear(self.markov_rank, self.vocab_size, bias=False)
```

Hidden states still come from one parallel backbone pass; only the token sampling walks the block sequentially, adding a rank-256 bias conditioned on the previous token. Gated and GRU-style recurrent variants also exist.

The second head is `AcceptRatePredictor` — a single `Linear(input_dim, 1)` that predicts per-position acceptance probability. This is the "confidence-scheduled" part of the name: at serving time a scheduler can truncate the speculative block as soon as predicted acceptance drops, rather than always drafting a fixed number of tokens. (The scheduling loop itself lives in SGLang; SpecForge trains and exports the head.)

Its objective is a three-term blend:

```python
loss = world_size * (
      self.dspark_ce_loss_alpha * ce_num            # 0.1 — CE vs ground-truth tokens
    + self.dspark_l1_loss_alpha * l1_num            # 0.9 — L1 distribution match
    + self.dspark_confidence_head_alpha * confidence_num   # 1.0 — BCE on the confidence head
) / global_loss_den
```

The weights tell you what the authors believe: CE is worth 0.1, matching the target's *distribution* in L1 is worth 0.9. And the L1 term connects straight back to the acceptance bound — `accept_probability = (1 - 0.5 * L1).clamp(0, 1)` is exactly the standard relationship between total variation distance and acceptance, and that detached quantity is what the confidence head is trained to predict. DSpark needs an extra captured tensor (`target_last_hidden_states`) to compute it.

### P-EAGLE: TTT without the sequential loop

P-EAGLE keeps EAGLE3's objective and attacks its *training cost*. Recall that TTT is seven dependent forwards — step $k$ cannot start until step $k-1$ finishes. P-EAGLE replaces the loop with a single forward over sampled (anchor, depth) pairs packed into one flat sequence:

![P-EAGLE replaces seven sequential TTT forwards with one packed forward over depth-sampled anchor-depth pairs](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-11.webp)

Depth 0 covers all positions with real tokens and real hidden states. For depth $d \geq 1$ it samples a decaying subset, `ratio = max(down_sample_ratio ** d, down_sample_ratio_min)` with defaults 0.8 and 0.2 — deeper rollouts get fewer training positions, mirroring the fact that deeper draft steps are reached less often. Depth>0 queries use `mask_token_id` plus a learned `mask_hidden` parameter in place of real target features.

The disambiguation is entirely in the mask:

```python
# specforge/algorithms/peagle/model.py
def peagle_mask_mod(_b, _h, q_idx, kv_idx):
    same_document   = document_ids[anchor_pos[q_idx]] == document_ids[anchor_pos[kv_idx]]
    same_rollout    = anchor_pos[q_idx] == anchor_pos[kv_idx]
    kv_depth0       = depth[kv_idx] == 0
    in_depth_order  = depth[q_idx] >= depth[kv_idx]
    is_anchor_causal = anchor_pos[q_idx] >= anchor_pos[kv_idx]
    return (is_not_padding & same_document
            & ((kv_depth0 & is_anchor_causal) | (same_rollout & in_depth_order)))
```

A query at (anchor $a$, depth $d$) sees all depth-0 positions with anchor $\leq a$, plus earlier depths of *its own* rollout only. The architecture is also deeper — four layers rather than one, with an EAGLE3-style double-width first layer and standard layers after it. The constraints are real: `flex_attention` only, and batch size must be exactly 1 per rank.

### EAGLE3.1: two flags, honestly

The README describes EAGLE3.1 as "feature-based autoregressive drafting with attention drift," which sounds like a new architecture. It is not. The example config documentation is explicit: "It remains registered as the `eagle3` strategy; EAGLE3.1 is a draft-model configuration variant, not a second runtime or launch path."

Diffing the two draft configs yields exactly two added keys: `"fc_norm": true` and `"norm_output": true`. The first inserts an independent RMSNorm on each of the three captured layer features before they are concatenated and projected:

```python
# specforge/modeling/draft/llama3_eagle.py — project_hidden_states
if self.fc_norm is not None:
    chunks = hidden_states.chunk(3, dim=-1)
    hidden_states = torch.cat([norm(chunk) for norm, chunk in zip(self.fc_norm, chunks)], dim=-1)
return self.fc(hidden_states)
```

This is a sensible fix for a real problem — activations at layer 1 and layer $N-4$ of a deep transformer have very different magnitudes, and concatenating them raw lets the larger-scale stream dominate the fused projection. But it is a normalization tweak, not a new lineage, and it is worth knowing that before you plan a migration around the version number.

## 8. Three topologies, and the target is never in the trainer

Now to the infrastructure half, which is where SpecForge earns its keep on large targets.

The single most important architectural fact: **the target model never runs inside the trainer process.** In offline mode it runs once in a preprocessing script and is gone. In online mode it runs in an external SGLang server. The only target-derived artifact living in the trainer is `TargetHead` — the target's `lm_head` weight, loaded from safetensors, frozen, in bf16, used to re-project stored hidden states into logits.

This is what makes training a drafter for a 671B target tractable on a single node of trainers: you are never holding 671B parameters and a draft optimizer in the same process.

![Three topologies: offline colocated, offline disaggregated and online disaggregated, differing only in where features are produced and how they reach the trainer](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-12.webp)

It is worth noting that the repo does not actually have a three-way mode enum. There are two orthogonal axes: the data mode is *derived* (`"offline" if data.hidden_states_path else "online"`), and `deployment.mode` is `local_colocated` or `disaggregated`. Three of the four combinations are legal — colocated online was removed deliberately, and validation says so plainly: "Online training is always disaggregated and captures through external SGLang servers; colocated online execution is intentionally absent."

### Offline: pay in disk, save in GPUs

```bash
# 1. build a conversation dataset (19 presets available)
python scripts/prepare_data.py --dataset ultrachat --split-eval

# 2. capture hidden states with a local SGLang engine
torchrun --nproc_per_node 8 scripts/prepare_hidden_states.py \
    --model-path Qwen/Qwen3-8B \
    --data-path  ./cache/dataset/ultrachat.jsonl \
    --output-path ./cache/hidden_states/qwen3-8b \
    --tp-size 8 --max-length 4096 --file-group-size 2000 --compress
```

Each sample becomes one file: `rows_{g}-{g+2000}/data_{idx}.ckpt`, holding `input_ids`, `loss_mask`, `hidden_state` (the post-norm final hidden) and `aux_hidden_state` (the three capture layers concatenated). The storage math is unforgiving. In bf16 you store $2 \times (3H + H) = 8H$ bytes per token, so for Qwen3-8B ($H = 4096$) a 4,096-token sample is about **128 MiB**. The LMSYS blog quotes roughly 12 TB for UltraChat plus ShareGPT combined. That is the trade: offline mode needs one GPU to train but a serious storage budget, and it is the reproducible choice because the features never change between runs.

Mitigations are everywhere in the code: gzip level 6 behind `--compress`, `torch.load(..., mmap=True)` when reading, one-sample-at-a-time writes through a bounded thread pool ("prioritizes minimal CPU RAM usage above all else"), and a resumability check that skips already-written indices so a crashed capture job restarts where it stopped.

### Online: pay in GPUs, save in disk

Online training keeps a patched SGLang server resident and streams features as they are produced. The producer sends prefill-only requests (`max_new_tokens: 1`, temperature 0) carrying a `spec_capture` payload, and — this is the part worth internalizing — **the tensors never pass through the producer process**:

> The server ... writes captured features straight into Mooncake in MooncakeFeatureStore's key layout. Tensors never pass through this process — the `/generate` response's `meta_info["spec_capture"]` carries only key/shape/dtype.

The server-side patch adds three flags (`--enable-spec-capture`, `--spec-capture-aux-layer-ids`, `--spec-capture-method`) and a new `spec_capture_sink.py` that writes one hard-pinned object per tensor into Mooncake at `{store_id}/{sample_id}/g{gen}/{name}`.

### The delivery guarantee

If features are produced by one process and consumed by another, you need exactly-once semantics — and you need them to survive a crash, because these runs take days.

![Exactly-once delivery: rank 0 is the single reader, and a feature is freed only after an optimizer-boundary durable ack](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-13.webp)

The chain is: the patched server writes tensors into Mooncake → metadata-only refs stream over an append-only fsynced JSONL channel → a `RefDistributor` on rank 0 (the only reader) leases refs and records them in a SQLite WAL ledger → whole optimizer windows are dispatched into one append-only inbox file per rank → each rank's `StreamingRefQueue` and `FeatureDataLoader` fetch tensors by ref → at the optimizer boundary `DPAckController` gathers sample ids and acks with `optimizer_durable=True` → only then are the Mooncake objects deleted.

Two design decisions in there are worth stealing for your own systems. First, Mooncake objects are **hard-pinned** on write, with SpecForge as "the lifetime authority" rather than Mooncake's LRU — because an LRU that evicts a committed-but-unacked feature produces a silent data loss that looks like a mysteriously worse model. Second, the ack is tied to the *optimizer* boundary, not to data loading; a feature that was loaded but whose step never made it to disk is correctly replayed on resume.

Backpressure is explicit: `producer_lease = 8`, in-flight watermarks of 192/256 refs, optional byte watermarks and a hard `feature_store_max_resident_bytes` cap that raises `MemoryError` rather than OOM-ing the box. There is even a validation that the low watermark is at least the consumer's global optimizer quantum (`dp_size * batch_size * accumulation_steps`), because otherwise producer and consumer can deadlock waiting for each other.

Online resume is **consumer-only**: rank 0 reconciles its retained SQLite ledger against the checkpoint's `global_step`, skips already-durable refs and requeues the unacked tail. The producer is never resumed — it is stateless by design.

### Launching it

The v0.2 CLI is a single typed entry point. There are no method-specific Python training scripts anymore; the README calls the removal "an intentional hard cutover."


```bash

specforge train --config ... --plan          # print the resolved launch plan as JSON
```

The launch planner is genuinely nice engineering. It produces a side-effect-free plan of one of four kinds: `worker` (we are already inside torchrun), `command` (self-launch torchrun), `supervisor` (spawn producer and consumer, propagate the first failure by SIGTERM-ing the peer's process group with a 30-second grace window), or `managed_supervisor` (also start `mooncake_master` and the patched SGLang capture servers, with readiness probes and preflight checks that the patch is actually applied). Unknown config fields and unknown override paths are errors, not warnings.

## 9. Parallelism and the memory tricks

The draft model is wrapped in FSDP over the world group with `use_orig_params=True` and `MixedPrecision(param_dtype=bf16, buffer_dtype=fp32)`. The default sharding strategy is `SHARD_GRAD_OP` — parameters gathered once, gradients and optimizer state sharded — which is the right default for a model this small.

A few decisions are worth calling out:

- **Trainer tensor parallelism is effectively dead.** It is forced to 1 for offline runs and for online consumers, with the config comment explaining that "target-model TP belongs to external or managed capture servers." TP survives only inside the offline capture script.
- **`NO_SHARD` is translated to DDP**, because PyTorch deprecated FSDP's NO_SHARD; the code notes this gives "replicated-param execution without per-block parameter all-gathers."
- **Frozen target tables are FSDP `ignored_modules`.** Sharding a frozen embedding or `lm_head` just adds all-gathers with no optimizer-memory saving.
- **Gradient accumulation uses `no_sync()`** on non-boundary micro-steps, so there is exactly one gradient collective per optimizer window. The Domino benchmark measured this directly: raising accumulation to 8 took a micro-step from 460 ms to about 280 ms.
- **USP sequence parallelism** (Ulysses all-to-all plus ring attention, via yunchang) is available for offline EAGLE3, requires batch size 1, and multiplies `accumulation_steps` by the SP size so one accumulation unit remains one complete logical sequence.

### The compact teacher

The nastiest memory problem in offline training is not the model — it is the teacher distribution. Projecting stored hidden states through the target's `lm_head` and upcasting to fp32 produces a `[B, S, vocab]` tensor. For one 4,096-token sample against a 151,936-token vocabulary that is about 2.4 GB *before* the softmax temporaries.

![The compact teacher streams the vocabulary in 32,768-column chunks instead of materializing a full-vocabulary fp32 logit tensor](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-14.webp)

The observation behind `compact_teacher` is that you only need three things from the teacher, and only two of them require the full vocabulary. The draft-space probabilities come from a sliced head; the normalizer and the argmax are computed by streaming:

```python
# specforge/core/compact_teacher.py
for start in range(0, vocab_size, chunk_size):          # DEFAULT_VOCAB_CHUNK_SIZE = 32768
    end = min(start + chunk_size, vocab_size)
    chunk_logits = F.linear(hidden, weight[start:end]).float()

    chunk_max = chunk_logits.max(dim=-1, keepdim=True).values
    new_max = torch.maximum(running_max, chunk_max)
    running_sumexp = running_sumexp * torch.exp(running_max - new_max) + torch.exp(
        chunk_logits - new_max
    ).sum(dim=-1, keepdim=True)
    running_max = new_max

    # Ascending scan + strict-greater update keeps the lowest global index on ties.
    chunk_val, chunk_idx = chunk_logits.max(dim=-1)
    chunk_idx = chunk_idx + start
    take = chunk_val > running_argval
    running_argmax = torch.where(take, chunk_idx, running_argmax)
    running_argval = torch.where(take, chunk_val, running_argval)

log_z = running_max + torch.log(running_sumexp)
```

Peak allocation drops from `[B, S, V]` to `[B, S, chunk]`. The teacher distribution is mathematically unchanged — and note the care taken over the argmax tie-break, using a strict-greater update on an ascending scan specifically so the result bit-matches `torch.argmax`. That is the kind of detail that separates a memory optimization you can trust from one that silently changes your training signal.

The DFlash family has an analogous trick on the *student* side: `checkpointed_chunk_reduce` slices the objective into `objective_chunk_blocks` anchor blocks (default 128) and wraps each in non-reentrant activation checkpointing, so vocabulary-sized logits for all 512 anchors never coexist in the autograd graph.

## 10. Export: the fail-silent boundary

Here is the failure mode that will cost you a day if nobody warns you.

A SpecForge checkpoint is **runtime state, not a model directory**. It contains `training_state.pt` (the filtered draft state dict, counters, and a run contract) plus per-rank `training_state_rank{r}.pt` files holding fp32 master weights, Adam state and RNG. You cannot point a server at it.

![From training state to a servable directory: the checkpoint filter, the deliberately dropped embedding, and the validated key set](/imgs/blogs/specforge-speculative-decoding-training-deep-dive-15.webp)

More importantly, SGLang's weight loading is **fail-silent**. If you hand it a directory with subtly wrong key names, it skips the keys it does not recognize and *the server still starts*. You get a running endpoint, no error in the logs, and an acceptance rate near zero — which looks exactly like a badly trained drafter. The exporter exists to make that impossible:

```python
# specforge/export/to_sglang.py
REQUIRED_KEYS = ("fc.weight", "norm.weight", "lm_head.weight", "t2d", "d2t")
```

It validates that all five are present, rejects any leftover `draft_model.` prefix, and deliberately drops `embed_tokens.weight` "exactly as the trainer-side checkpoint filter does" because serving reuses the target's embedding table.

```bash
# EAGLE3 -> a directory SGLang can load directly
specforge export --to sglang \
    --checkpoint ./outputs/qwen3-8b-eagle3/qwen3-8b-eagle3-latest \
    --draft-config configs/qwen3-8b-eagle3.json \
    --output-dir ./exports/qwen3-8b-eagle3-sglang

# DFlash-family and P-EAGLE go through the HF exporter instead
specforge export --to hf \
    --checkpoint ./outputs/qwen3-8b-dflash/qwen3-8b-dflash-latest \
    --embedding-source Qwen/Qwen3-8B \
    --output-dir ./exports/qwen3-8b-dflash-hf
```

Then serving is stock SGLang — no plugin, no conversion:

```bash
python3 -m sglang.launch_server \
    --model <target-model-path> \
    --speculative-algorithm EAGLE3 \
    --speculative-draft-model-path ./exports/qwen3-8b-eagle3-sglang \
    --speculative-num-steps 3 \
    --speculative-eagle-topk 1 \
    --speculative-num-draft-tokens 4
```

For DFlash the flags differ (`--speculative-algorithm DFLASH`, `--speculative-dflash-block-size 16`). Add `--tp`, `--ep` and `--mem-fraction-static` as memory requires.

The three speculative knobs deserve a note, because they interact. `--speculative-num-steps` is how many draft steps run per round; `--speculative-eagle-topk` is the tree branching factor; `--speculative-num-draft-tokens` is how many tokens the tree ultimately submits for verification. Across the published SpecBundle dashboard, the best configuration is almost always `steps=7, topk=1, num_draft_tokens=8` — that is, a *deep chain* rather than a wide tree. Wide trees pay verification cost on tokens that mostly get rejected. Qwen3-Next is the exception where `topk=4` wins.

## 11. What the docs actually recommend

Reading the training guide and the recipe configs together gives a fairly clear picture of the house style.

### Regenerate your training data with the target model

This is the highest-leverage practice in the whole framework and the least obvious. Rather than training the drafter on the original human/GPT-authored answers, you regenerate the assistant responses using **your target model**, then train on those. The docs are careful about why:

> According to the EAGLE1 paper, the EAGLE method is not very sensitive to the dataset quality... However, if you are looking for optimal performance in the production environment, it is recommended to regenerate.

The logic is that the drafter's job is not to be a good language model — it is to *imitate this specific target*. Training it on text the target would never produce teaches it a distribution it will be penalized for at verification time. Every production SpecBundle checkpoint is trained on regenerated data, and the docs blame low acceptance in other open checkpoints on "small or curated datasets."

```bash
# serve the target for generation
python3 -m sglang.launch_server --model-path Qwen/Qwen3-8B \
    --cuda-graph-max-bs 128 --dtype bfloat16 --mem-fraction-static 0.8

# regenerate answers (data-parallel: pass --server-address multiple times)
python scripts/regenerate_train_data.py \
    --model Qwen/Qwen3-8B \
    --concurrency 128 --max-tokens 4096 --temperature 0 \
    --server-address localhost:30000 \
    --input-file-path  ./cache/dataset/sharegpt.jsonl \
    --output-file-path ./cache/dataset/sharegpt-regen.jsonl \
    --expect-non-reasoning
```

The script has strict accounting (`success + error + skipped == input`), refuses to overwrite, and writes `_error.jsonl` / `_skipped.jsonl` sidecars. For reasoning models, `--reasoning save` stores `reasoning_content` (requires `--reasoning-parser qwen3` on the server) while `--reasoning disable` forwards `chat_template_kwargs.enable_thinking=false`.

### Hyperparameters: schema defaults versus real recipes

The schema defaults and the shipped recipes disagree, and the recipes are what produced the published checkpoints.

### Evaluate the right thing

Training-time acceptance is a proxy. The docs say so explicitly: it is "not the same measurement as serving-time accepted length." The default best-checkpoint metric is `eval/simulated_acc_len`, and offline evaluation requires both `data.eval_hidden_states_path` and `training.eval_interval` (online evaluation is unsupported and fails validation).

Real evaluation runs against a server:

```bash
python benchmarks/bench_eagle3.py \
    --config-list "1,0,0,0" "8,3,1,4" "8,7,1,8" \
    --benchmark-list mtbench:80 gsm8k:200 humaneval:164 math500:200
```

The `1,0,0,0` row is the baseline with speculation disabled — **always include it**. The docs are emphatic about comparison hygiene: same target revision, tokenizer, template, prompts, sampling params, output length, hardware, TP degree and concurrency, and never compare absolute throughput between the matrix runner and the existing-server runner. Also worth knowing: "The EAGLE3 HumanEval and MBPP scorers execute model-generated Python" — run them somewhere isolated.

Measurement windows matter too. The Domino benchmark notes that supply is bursty by roughly 1.8x and recommends measuring for at least 390 seconds; short windows produce numbers you will later fail to reproduce.

## Case studies from production

These are drawn from the repository's own benchmark notes, patch inventory and code comments — every one is a real, documented finding rather than a hypothetical.

### 1. The capture server hoarding KV cache it never used

A Domino run on a single 8×H200 node was producing 28.8 samples/s and the trainers looked starved. The capture server was configured the way you would configure a *serving* server: `--mem-fraction-static 0.85`, reserving roughly 126 GB for KV cache. But a capture server never decodes — it prefills a prompt, emits one token, and writes hidden states. Almost all of that KV cache was dead weight. Dropping to `--mem-fraction-static 0.5` (about 78 GB) freed memory and was one component of a configuration-only tuning arc that took the same hardware from 28.8 to **50.1 samples/s, a 74% gain**, with no code change. The lesson generalizes: a model server doing prefill-only capture has a completely different memory profile from one serving traffic, and inherited serving defaults are actively wrong.

### 2. The GPU split seesaw

The same investigation asked the obvious next question: if the capture server is the bottleneck, add another one. With 8 GPUs, one server plus seven trainers gave 50.1 samples/s. Two servers plus six trainers gave **39.0-39.5 samples/s** — meaningfully worse. The measured per-trainer demand was about 7 samples/s, and one server could supply 52-57 samples/s. So a single server was already over-provisioned for seven trainers; converting a trainer into a second server removed demand-side capacity to solve a supply-side problem that did not exist. The regime flipped from supply-bound to demand-bound. Measure both sides before rebalancing, because the intuition "add more of the thing that looked busy" is wrong half the time.

### 3. Radix cache poisoning the captures

SGLang's radix cache is a large win for serving: identical prefixes reuse cached KV. For feature capture it is a correctness hazard — if a prompt's prefix is served from cache, the forward pass that would have produced the hidden states never runs, and you capture nothing (or worse, something partial). The naive fix is to disable the radix cache entirely. SpecForge instead attaches a unique `extra_key` UUID to every capture request, so each training sample is forced through a full prefill while the radix cache stays enabled for everything else. It is a much cheaper fix than a global flag, and it keeps the capture server usable for other work.

### 4. Chunked prefill silently dropping hidden rows

The server-side patch refuses to start if `--enable-spec-capture` is used without `--chunked-prefill-size -1`. The reason, straight from the code: "Capture needs single-pass prefill; chunking would drop all but the final chunk's hidden rows." With chunked prefill, a long prompt is processed in pieces and only the last chunk's hidden states survive into the capture path — so you would silently train on features for the tail of every conversation and nothing else. This would not crash. It would not warn. It would just produce a drafter that is inexplicably bad on long contexts. Turning it into a hard startup error is the correct engineering call.

### 5. Mooncake's LRU versus the training ledger

Mooncake is an object store with LRU eviction, which is exactly right for a cache and exactly wrong for a delivery queue. If a captured feature is evicted after the ref was published but before the trainer consumed it, the trainer sees a missing object for a sample it was told exists. SpecForge writes every object with `with_hard_pin=True` and `replica_num=1`, and the comment states the ownership rule plainly: SpecForge, "not Mooncake's LRU," is the lifetime authority. Objects are removed explicitly, only after the optimizer step that consumed them is durable. The general principle: when you borrow a cache as a queue, you must take over its eviction policy or you have built a lossy queue.

### 6. The exporter that exists because loading is fail-silent

The single most instructive design decision in the repo. SGLang loads draft weights by name and **skips unrecognized keys without raising**. A checkpoint saved with a `draft_model.` prefix, or missing `t2d`/`d2t`, produces a server that boots cleanly and speculates terribly. Because the failure looks identical to "the model just isn't very good," teams burn days retraining. The exporter's answer is to validate the exact five-key contract at export time and refuse to write a directory that would fail silently later. If you build any system that hands weights to a loader you do not control, put the assertion on *your* side of the boundary.

### 7. Compiled block-mask creation that was slower than not compiling

In the FlexAttention path there is a deliberate branch: for `q_len <= 128` the code uses the *uncompiled* `create_block_mask` and `flex_attention`, with a TODO referencing [pytorch#160018](https://github.com/pytorch/pytorch/issues/160018). Compiled block-mask creation is broken or pathologically slow for short sequences, so the optimization is switched off precisely where it would hurt. Alongside it: `dynamo.config.recompile_limit = 64`, because the mask shape changes on every one of the seven TTT steps and the default recompile budget is exhausted immediately. Both are the sort of thing you only find by profiling, and both would look like arbitrary magic numbers to a reader who skipped the comments.

### 8. bf16 buffers, fp32 buffers, and a broken A/B comparison

The v0.2 rewrite standardized on fp32 buffers under FSDP; the removed EAGLE3 and DFlash scripts used bf16 buffers. The migration doc calls this out specifically because anyone comparing a v0.2 run against a v0.1 baseline bit-for-bit will find a discrepancy that has nothing to do with the change they are investigating. This is the quiet tax of any framework rewrite: the numbers move for reasons unrelated to your experiment, and unless someone writes it down, the next engineer spends a week bisecting.

### 9. A chat template anchored to text that never appears

The template registry has a pointed comment for DeepSeek-V2-Lite: it must use plain-text headers "or the assistant loss mask will be anchored at text that never appears." Loss masking works by regex-matching the assistant span in the rendered conversation. If the template's header string does not literally occur in the rendered text, the regex matches nothing, `loss_mask` is all zeros, and training proceeds with a loss computed over no supervised positions. The model trains on nothing and the loss curve looks suspiciously flat. Chat-template bugs are the single most common silent failure in drafter training, and they are invisible unless you inspect the mask on a real sample before launching a multi-day run.

### 10. Parity to 1e-6 as a release gate

Before shipping the disaggregated path, the team ran the same Qwen2.5-7B training in colocated and disaggregated topologies and compared metrics step by step. At step 20 the acceptance rate was 0.0013300 in both; `ploss` was 5.386736 versus 5.386740. At step 180, acceptance was 0.0337013 versus 0.0336982. Agreement to roughly 1e-6, published as a benchmark doc. This is what it looks like to take a refactor seriously: the claim "the disaggregated path is equivalent" is not an assertion in a README, it is a reproducible measurement with a commit hash attached. If you are building a distributed training path, steal this practice — numerical parity against a simpler topology is the only convincing evidence that your plumbing is not silently changing the math.

## When to reach for SpecForge

**Reach for it when:**

- You serve on SGLang and want speculative decoding on a model with no published draft checkpoint — a fine-tune, an internal model, or a new release.
- Your traffic is **latency-sensitive at low-to-moderate batch size**: interactive chat, coding assistants, agent loops that decode short bursts. This is where the free-FLOPs argument holds.
- Your output is structured — code, math, JSON, tool calls. Accepted length runs 4.3-5.6 there versus 2.8-3.0 on open conversation, and the speedup follows.
- You have a domain-specific corpus. A drafter trained on your traffic beats a general checkpoint, and the vocabulary mapping alone is worth the retraining.
- You already run a large target with TP/EP and cannot fit it beside a trainer — the disaggregated design was built for exactly this.
- You want the lossless guarantee: this is a throughput optimization with **zero output-quality risk**, which makes it unusually easy to justify.

**Skip it when:**

- You are saturating the GPU at high sustained batch size. The FLOPs you would spend on drafting are no longer spare, and speculation can *reduce* throughput. Measure before assuming.
- A published SpecBundle checkpoint already covers your exact target and domain — start there and only train when you have measured a gap.
- You cannot afford the capture cost: offline mode wants roughly 128 MiB per 4k-token sample (about 12 TB for UltraChat plus ShareGPT), and online mode wants a dedicated capture GPU for the run's duration.
- Your workload is prefill-dominated — long documents in, short answers out. Speculative decoding accelerates decode; it does nothing for prefill.
- You need VLM support today. Vision configs exist in the tree, but the docs state the unified runtime "currently accepts text inputs only."
- You are not on SGLang. The whole value proposition is the zero-porting boundary; on another serving stack you inherit the export problem the framework was designed to eliminate.

One closing thought about where this sits in the ecosystem. The interesting thing about SpecForge is not any single algorithm — EAGLE3 is a published paper, DFlash is a published paper. It is that the framework treats *the boundary between training and serving* as the primary engineering artifact. The capture contract is validated. The export key set is asserted. The disaggregated path is proven equal to the colocated one to six decimal places. The patched server refuses to start in a configuration that would silently corrupt captures.

> Most ML infrastructure fails at the seams, not in the algorithms. The seams are where nobody owns the invariant.

That is the part worth copying even if you never train a draft model.
